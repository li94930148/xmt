# XMT 内容生产中台 — 系统化升级计划

基于对项目代码的全面审计（前端 28 页面/31 组件/18 API 模块、后端 20 路由/25+ 数据库表、共享类型 30 实体），结合需求文档的八大维度，制定以下分阶段升级计划。

---

## 一、项目现状审计摘要

### 已有且完善的
- 完整内容工作流：选题 → 创作 → 拍摄 → 发布，含状态机校验
- RBAC 权限体系（数据库驱动，4 角色 22 权限）
- Socket.IO 实时通信（消息推送、多房间广播）
- Tiptap 富文本编辑器（批注、目录、版本历史）
- 数据分析仪表盘 + 抖音数据抓取
- 成就系统 + 番茄钟 + 灵感池
- 深色/亮色主题 + 骨架屏 + 错误边界
- 全局命令面板 (Ctrl+K) + 桌面通知

### 明显问题（需优先修复）
1. **API 响应格式不统一** — `response.ts` 已定义规范但仅 `topics.ts` 使用，其余 19 个路由直接 `res.json()`
2. **双权限体系并行** — 44 处路由用旧的 `requireRole`（硬编码），仅 7 处用新 RBAC `requirePermission`
3. **两套编辑器共存** — Tiptap (`editor/Editor.tsx`) 和 Quill (`RichTextEditor.tsx`) 并存，Topics/AddTopic 仍用 Quill
4. **类型定义不完整** — `UserRole` 缺 `editor`；`Production/Shooting/Publishing.status` 用 `string` 非枚举；缺 `Role`/`Permission` 实体类型
5. **API 请求封装重复** — 无 axios 实例，每个 `src/api/*.ts` 重复 `getAuthHeader()` + `fetch` 模板代码
6. **Mock API 残留** — `AnnouncementBoard.tsx`、`PomodoroTimer.tsx` 内部定义了 mock 函数而非用真实 API
7. **无测试框架** — 无 jest/vitest，无任何测试文件
8. **无锁文件** — `package-lock.json` 缺失
9. **错误响应泄露内部信息** — 部分路由返回原始 Error 对象
10. **备份绕过 API 层** — `BackupPage.tsx` 直接用 `fetch` 而非 `api/backup.ts`

## 九、验证方式

每轮完成后执行：
1. `npm run check` — TypeScript 类型检查通过
2. `npm run lint` — ESLint 检查通过
3. `npm run build` — 生产构建成功
4. `npm run dev` — 开发服务器正常启动
5. 手动测试关键流程：登录 → 创建选题 → 状态流转 → 发布
