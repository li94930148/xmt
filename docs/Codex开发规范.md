# Codex开发规范

## 目的

本规范用于约束 Codex 在 XMT 项目中的协作方式，确保所有修改可追踪、可验证、可回顾，并尽量降低对现有业务的扰动。

本项目当前处于“内容生产中台治理期”，同时正在向“自有品牌中台”升级。后续还将进一步抽象为可复制、可扩展的行业通用系统。因此，Codex 在执行任务时，既要关注当前代码真实情况，也要关注未来平台化沉淀价值。

## 1. 先分析后执行

1. 接到任务后，必须先分析项目结构、相关模块、现有文档和真实实现。
2. 在没有足够上下文前，不得直接修改代码、数据库或配置。
3. 涉及业务链路、权限、数据库、实时通信、部署时，必须先确认影响面。

## 2. 修改前输出方案

1. 在执行任何修改前，先说明准备修改的范围、思路和预期影响。
2. 方案应尽量明确到目录、文件、模块和风险点。
3. 如果存在多个可行方案，优先给出最小修改方案。

## 3. 修改后输出变更报告

1. 完成修改后，应输出简明变更报告。
2. 报告至少包含：
   - 修改了什么
   - 为什么这样改
   - 是否执行了验证
   - 是否存在遗留风险或后续建议

## 4. 优先最小修改原则

1. 优先选择对现有系统扰动最小的修改方式。
2. 不因顺手优化而扩大改动范围。
3. 不在一次任务中混入无关重构、命名清洗或风格统一。
4. 能局部修复时，不做全局替换。

## 5. 禁止无关代码修改

1. 只处理当前任务明确要求的内容。
2. 不得修改与本次目标无关的业务代码、样式、配置或脚本。
3. 遇到历史问题、潜在优化点或过时代码时，可以记录说明，但不能擅自一并修改。

## 6. 禁止大规模扫描

1. 阅读代码时应先缩小范围，再进入具体文件。
2. 优先查看入口文件、相关模块和必要配置。
3. 避免无目标地全仓扫描，减少无效上下文和误判风险。

## 7. 优先读取 docs 文档

1. 开始任务前，优先查看 `docs/` 下已有文档。
2. 如果文档与代码不一致，以当前代码和数据库真实状态为准。
3. 如确认文档过时，应在结果中明确标注，并在允许范围内更新文档。

## 8. 文档与代码一致性原则

1. 文档必须尽量基于真实代码结构、真实页面、真实数据库编写。
2. 现状与规划必须明确区分，禁止把规划写成已实现。
3. 涉及数据库设计时，优先以实际数据库和初始化脚本共同校验。
4. 涉及平台方向时，允许写入品牌化、平台化、行业化目标，但必须明确属于规划内容。

## 9. 品牌化与平台化写作原则

1. 当前 XMT 不只是内部工具，而是未来自有品牌中台的原型系统。
2. 文档中应逐步沉淀可复用的方法论、模块边界和平台能力，不只记录单次需求。
3. 在不脱离真实代码的前提下，优先提炼未来可通用于多品牌、多团队、多行业的能力：
   - 内容流程配置
   - 权限体系
   - 实时协作
   - 模板能力
   - 数据分析与导出
   - 系统设置与品牌配置

## 10. 风险控制原则

1. 涉及认证、权限、数据库、备份、部署、实时通信时，提高谨慎级别。
2. 高风险操作前必须先说明影响面。
3. 未经明确授权，不执行删除、重置、回滚或批量替换类高风险操作。

## 11. 文档类任务特殊要求

1. 文档建设任务只允许创建或更新文档，不触碰业务实现。
2. 文档应优先服务于后续治理、协作、品牌升级和平台复制。
3. 如果发现现有文档乱码、过时或口径不一致，应优先统一文档质量。

## 12. 推荐执行顺序

1. 阅读 `docs/` 和相关入口文件
2. 分析任务涉及模块
3. 输出执行方案
4. 实施最小范围修改
5. 进行必要验证
6. 输出变更报告

## 13. 本项目协作要求

针对 XMT 项目，Codex 额外遵守以下要求：

1. 优先保持前后端同仓结构和业务链路说明的清晰性。
2. 涉及路由、数据库结构、权限、实时事件时，必须同时考虑前后端联动影响。
3. 遇到历史遗留模块，要区分“当前主路径”和“兼容保留结构”。
4. 所有文档输出都应服务于一个长期目标：
   - 先把 XMT 打造成我们自己的品牌中台
   - 再把它抽象成可复制的行业通用系统

## 后续开发策略：中等规模、分阶段、可验证的结构治理

当前项目已经完成安全治理、权限治理、系统设置治理、页面基础框架统一、公共组件规范补齐和编辑器统一入口初步建设。后续开发策略从“严格最小改动”调整为“中等规模、分阶段、可验证、可回滚的结构治理”。

这意味着后续不再过度限制为“只能最小改动”，但也不允许无计划大改。每次任务仍必须限制在当前任务主题内，先明确边界，再分阶段执行和验证。

### 1. 允许的修改力度

后续允许在当前任务主题内进行：

1. 中等规模组件拆分。
2. 同一功能链路内多文件修改。
3. 页面内部状态逻辑整理。
4. 业务组件提取。
5. 重复代码收敛。
6. 文件结构小幅调整。
7. 模块内可维护性优化。

### 2. 仍然禁止的行为

未经明确确认，仍然禁止：

1. 一次性重写整个项目。
2. 一次性重构多个无关模块。
3. 修改数据库结构。
4. 修改接口协议。
5. 修改权限模型。
6. 修改内容数据格式。
7. 引入新框架。
8. 删除旧组件。
9. 改变业务流程。
10. 实现协同编辑。
11. 改动部署方式。

### 3. 中等规模任务执行前必须输出方案

执行前必须输出：

1. 准备修改哪些文件。
2. 每个文件准备改什么。
3. 是否影响业务逻辑。
4. 是否影响接口。
5. 是否影响数据库。
6. 是否影响权限。
7. 是否影响用户操作流程。
8. 风险点。
9. 回滚方式。
10. 验证方式。

等待确认后再执行。

### 4. 执行完成后必须输出结果

执行完成后必须输出：

1. 实际修改文件。
2. 实际修改内容。
3. 是否与方案一致。
4. 是否有额外改动。
5. 是否影响业务逻辑。
6. 是否影响接口、权限或数据库。
7. `npm run check` 结果。
8. `npx tsc --noEmit` 结果。
9. 后续建议。

### 5. 核心原则

后续项目进入中等规模、分阶段、可验证、可回滚的结构治理阶段。

不再过度限制为“只能最小改动”，但也不允许无计划大改。所有中等规模治理都必须服务于明确的模块边界、业务链路或长期架构收益。
## Workflow Engine 修改规范

当前仓库的 Workflow Engine 已进入 `v2_strict` 状态。后续涉及工作流、审批、流程模板、状态流转、WorkflowDesigner 的任务，必须优先按当前真实实现分析，不得再将系统描述为“仅提示、不阻断”的 Shadow-only 或 Partial-only 状态。

### 1. 修改前必须确认运行模式

执行任何 Workflow Engine 相关任务前，必须确认：

1. `WORKFLOW_ENGINE_MODE` 当前值。
2. 本次修改是否影响 `shadow`、`v2_partial`、`v2_strict` 任一模式。
3. 是否会改变审批 API 的阻断条件。
4. 是否会改变 WorkflowDesigner 的非法节点保存策略。
5. 是否会改变 `topic.status` 或审批结果。

### 2. 严格控制高风险改动

以下改动必须先输出方案并等待确认：

1. 修改 `shared/workflow/policy.ts` 中的状态转换规则。
2. 修改 `canApproveNode`、`canTransition`、`canConfigureTransition` 的判断口径。
3. 修改 `api/services/workflow_decision_engine.ts` 的 `allowed`、`shouldWarn`、`confidence` 或推荐逻辑。
4. 修改审批 API 中 strict gate 的返回条件。
5. 修改 `workflow_shadow_logs` schema 或删除历史观测数据。
6. 修改 WorkflowDesigner 对非法节点的阻断、提示或自动应用建议行为。

### 3. 文档口径要求

Workflow Engine 文档必须区分四类能力：

1. Shadow Logging：记录运行上下文。
2. Analytics：聚合观测数据。
3. Decision：生成风险、建议和解释。
4. Strict Control：对非法审批或非法节点配置执行阻断。

如果代码已经启用 `v2_strict`，文档不得写成“不会阻断审批流程”。正确表述应为：系统保留 Shadow / Analytics / Decision 可观测链路，并在 strict mode 下对策略不通过的审批和节点配置执行强约束。

### 4. 验证要求

Workflow Engine 相关修改完成后，除非任务明确只做只读分析，否则必须运行：

1. `npm run check`
2. `npx tsc --noEmit`

如涉及审批 API 行为，还应在结果中说明是否影响生产审批、是否可回滚、是否改变数据库结构、是否改变历史数据语义。
