# Codex开发规范

## 目的

本规范用于约束 Codex 在 XMT 项目中的协作方式，确保所有改动可控、可追踪、可回溯，并尽量减少对现有业务的扰动。

## 1. 先分析后执行

1. 接到任务后，必须先分析项目结构、相关模块和已有实现。
2. 在没有足够上下文前，不得直接修改代码、配置或数据。
3. 对业务链路不清楚时，应优先阅读现有代码和文档，再决定执行方式。

## 2. 修改前输出方案

1. 在进行任何代码、配置或数据相关修改前，应先说明准备修改的范围和思路。
2. 方案应尽量明确涉及的目录、文件和预期影响。
3. 若存在多种可行方案，应优先给出最小改动方案。

## 3. 修改后输出变更报告

1. 完成修改后，应输出简明变更报告。
2. 变更报告至少应说明：
- 修改了什么
- 为什么这样改
- 是否执行了验证
- 是否存在遗留风险或后续建议

## 4. 优先最小修改原则

1. 优先选择对现有系统扰动最小的改法。
2. 不因顺手优化而扩大改动范围。
3. 不在一次任务中混入无关重构、命名清洗或风格统一。
4. 能局部修复时，不做全局替换。

## 5. 禁止无关代码修改

1. 只处理当前任务明确要求的内容。
2. 不得修改与本次目标无关的业务代码、样式、配置或脚本。
3. 遇到历史问题、潜在优化点或过时代码时，可以记录说明，但不能擅自一并修改。

## 6. 禁止大规模扫描

1. 读取代码时应先缩小范围，再进入具体文件。
2. 优先查看入口文件、相关模块和必要配置。
3. 避免无目标地全仓大规模扫描，减少无效上下文和误判风险。

## 7. 优先读取 docs 文档

1. 开始任务前，优先查看 `docs/` 下已有文档。
2. 若文档与实际代码不一致，应以当前代码为准，并在结果中注明文档可能过时。
3. 新增能力、架构变动或流程约束时，应同步补充或更新相关文档。

## 8. 文档与代码一致性原则

1. 文档必须尽量基于真实代码结构编写，避免把规划写成现状。
2. 描述当前能力时，应明确哪些已经实现，哪些仍是计划。
3. 涉及架构、接口、数据库时，应以实际仓库内容为主要依据。

## 9. 风险控制原则

1. 涉及认证、权限、数据库、备份、部署和实时通信时，应提高谨慎级别。
2. 高风险改动前必须先确认影响面。
3. 未经明确授权，不执行删除、重置、回滚或批量替换类高风险操作。

## 10. 执行顺序建议

推荐遵循以下工作顺序：

1. 阅读 `docs/` 和相关入口文件
2. 分析任务涉及模块
3. 输出执行方案
4. 实施最小修改
5. 进行必要验证
6. 输出变更报告

## 11. 本项目协作要求

针对 XMT 项目，Codex 应额外遵守以下要求：

1. 优先保持前后端同仓结构的清晰性。
2. 涉及路由、数据库结构、权限、实时事件时，必须同时考虑前后端联动影响。
3. 涉及文档建设任务时，只创建或更新文档，不触碰业务实现。
4. 对已确认过时但未删除的文件，应在分析中标记为“忽略参考”，避免误用。
