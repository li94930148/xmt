# P1-004E 前置评估

评估日期：2026-06-24

## 评估边界

本轮只核对评论删除、资源库、排期日历与权限缓存刷新策略，不新增权限点，不修改数据库、默认角色权限种子或 Workflow Engine v2 语义。

## 当前真实状态

### 评论删除

`DELETE /api/workflow/comments/:id` 当前只校验 `comment:delete`，不查询评论创建者，也没有 owner 分支。因此 `comment:delete` 的准确语义是“删除任意流程评论”，普通用户不能仅凭本人身份删除自己的流程评论。

后续如产品确认需要普通协作者撤回本人评论，建议评估：

1. `comment:delete_own`：删除本人流程评论，并明确撤回时限与审计留痕。
2. `comment:delete_any`：删除任意流程评论，承接当前 `comment:delete` 的管理语义。
3. 新旧权限的兼容迁移与默认角色矩阵应先评审，本轮不新增上述权限点。

### 资源库

资源库当前是登录后可创建，查看、更新、删除按 uploader owner 或 privileged user 控制。该口径当前没有明显越权缺口，应继续保留。

平台化前建议评估 `resource:view`、`resource:create`、`resource:update`、`resource:delete`。这些候选权限负责能力入口，owner / privileged scope 继续负责数据范围，两者不能互相替代。

### 排期日历

排期日历当前是登录后可查看本人事件和相关选题、可创建事件，更新与删除按事件 owner 或 privileged user 控制。该口径当前没有明显越权缺口，应继续保留。

多人排期产品化前建议评估 `calendar:view` 与 `calendar:manage`，并继续叠加事件 owner、选题 scope 和 privileged scope。

### 后端权限缓存

后端权限缓存 TTL 为 5 分钟。`api/middleware/permissions.ts` 已定义并导出 `clearPermissionCache`，但当前角色权限更新、用户角色分配及用户角色字段更新均未调用它。

因此角色或权限映射变更后，已缓存用户的后端权限判断最长可能延迟约 5 分钟。

### 前端权限状态

`/api/permissions/my` 当前直接读取数据库，不使用上述后端缓存。但前端 `usePermission` 把权限保存在各 Hook 实例的本地状态中，只在 `user` 变化时自动获取。

Hook 虽暴露 `refreshPermissions`，管理操作没有通知被调整用户的在线客户端，各 Hook 实例之间也不共享刷新结果。因此前端菜单、路由和按钮可能保持旧权限，直到对应实例重新挂载、用户重新登录或显式刷新。

## 最小执行方案

1. 在现有用户角色变更成功后调用 `clearPermissionCache(userId)`。
2. 角色权限批量变更后，清理该角色关联用户的缓存；若暂不实现精确定位，可保守调用全量清理。
3. 将前端权限收口为单一共享状态，避免多个 `usePermission` 实例各自持有快照。
4. 明确刷新触发：登录、角色/权限变更通知、窗口重新聚焦，或权限接口返回 403 后刷新一次。
5. 评论、资源、日历候选权限只完成产品语义、默认角色矩阵和兼容迁移评审；待单独批准后再新增，不与缓存修复绑在一起。

## 本轮结论

P1-004E 已完成前置评估和方案冻结。本轮不改业务代码，不新增权限点，不改数据库。
