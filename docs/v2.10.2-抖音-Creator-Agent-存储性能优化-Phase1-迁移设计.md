# XMT v2.10.2 抖音 Creator Agent 存储性能优化：Phase 1 迁移设计

## 1. 本阶段结论

本阶段只完成代码、数据和查询计划审计，不执行生产数据库迁移，不删除或改写任何历史数据。

v2.10.2 不应被实现为“把 28 条修成 12 条”的一次性清洗。正确改造方向是：

1. 用一个严格、平台可扩展的 `DouyinWorkInput` 作为 Agent 到服务端的唯一作品输入。
2. 将 `creator_content_items` 作为跨平台作品身份主表，将 `douyin_works` 作为抖音业务读模型；两表必须在同一事务中从同一输入写入，并通过 `content_id` 建立一对一关系。
3. 作品静态元数据、最新指标、指标时序和原始响应分层保存。
4. `discover`、`metrics_refresh`、`full_snapshot` 使用不同写入语义，禁止发现任务以零值覆盖指标。
5. v2.10.1 无版本契约继续走兼容适配器；只有 `contract_version = "2.10.2"` 才走严格新链路。
6. 所有清理均拆成“标记/预演/审批/执行”四步；Phase 1 到 Phase 4 均不得物理删除历史数据。

## 2. 当前实现证据

### 2.1 双写链路不是同一数据源

- `api/services/creatorSyncV291.ts` 先把 `payload.contents` 写入 `creator_content_items`，再调用 `persistNormalizedDouyinSync` 写 `douyin_works`。
- 通用链路直接信任 Agent 的 `contents`；抖音链路重新从 `raw_records` 和 `contents` 中寻找带统计字段的作品。两者过滤条件不同，因此不可能保证 ID 集合一致。
- 当前数据库只读副本统计：
  - `creator_content_items`：28 条；
  - `douyin_works`：12 条；
  - 两表抖音作品 ID 交集：0；
  - 仅通用表存在：28；仅抖音表存在：12。

这不是简单的重复插入问题，而是两个互不等价的作品识别器分别落库。

### 2.2 Agent 解析器会把非作品对象当成作品

`agent/core/collector/douyin/parser/common.ts` 会递归遍历所有网络响应对象，只要对象具有通用 `id` 和 `title/name` 就视为作品。它没有限制响应端点、数组容器或实体类型。

合成回归输入 `12 aweme + 12 music + 7 module`，当前实现实际输出 31 条：12 个作品、12 个音乐对象、7 个模块对象。数据库中也已出现 `@douyin-creator-pc/common:*` 形式的组件 ID。

### 2.3 长 ID 在进入字符串链路前已可能损失精度

`agent/core/network/response-collector.ts` 使用 `response.json()`。如果上游把 ID 作为 JSON number 返回，JavaScript 在 `String(...)` 之前就已按 IEEE-754 解析并损失精度：

```text
7663799549412758193 -> 7663799549412759000
```

因此“数据库列为 TEXT”不足以保证全链路字符串。采集入口必须保留原始响应文本，并用大整数安全 JSON 解析策略把 ID 字段直接解析成字符串；契约校验必须拒绝 number 类型 ID。

### 2.4 JSON 存储放大已经出现

只读副本中的当前占用（SQLite `length()` 逻辑字节，不含页和索引开销）：

| 区域 | 行数 | JSON 字节 |
| --- | ---: | ---: |
| `creator_content_items.raw_json` | 28 | 322,913 |
| `creator_content_metrics.raw_json` | 56 | 654,268 |
| `creator_api_raw_records.response_json` | 187 | 367,540 |

28 条通用作品对应三处约 1.28 MB JSON。按当前形态扩展到 50 万作品和每日指标不可接受。

### 2.5 当前代码含未经审批的删除

- `api/services/douyinDataCenter.ts` 每次同步都会执行按标题删除 `creator_content_items` 的 SQL。
- `docs/migrations/v2.10.1.sql` 末尾也包含同类 `DELETE`。

v2.10.2 实施前必须移除运行时删除；历史污染数据只能先进入隔离清单，待 Phase 5 单独审批。

## 3. 目标数据流和职责

```text
Agent 2.10.1（无 contract_version）
  -> LegacyPayloadAdapter
  -> DouyinWorkInput[]

Agent 2.10.2
  -> 严格契约校验
  -> DouyinWorkInput[]

DouyinWorkInput[]
  -> 同一数据库事务
     -> creator_content_items（跨平台身份与共享元数据）
     -> douyin_works（抖音状态、最新指标缓存和采集调度）
     -> douyin_work_metrics（仅 metrics_refresh/full_snapshot 追加）
```

### 3.1 `DouyinWorkInput` 契约

```ts
type CollectionMode = 'discover' | 'metrics_refresh' | 'full_snapshot';

type DouyinWorkInput = {
  aweme_id: string;               // 必须是非空字符串，禁止 number
  title?: string;
  cover_url?: string;
  publish_time?: string | null;   // ISO-8601
  video_url?: string;
  status?: string;
  metrics?: {
    play_count: number;
    like_count: number;
    comment_count: number;
    share_count: number;
    collect_count: number;
    interaction_rate?: number;
    captured_at: string;
  };
};
```

上传包必须在加密载荷中包含：

```json
{
  "contract_version": "2.10.2",
  "collection_mode": "discover",
  "snapshot_id": "UUID/ULID 字符串",
  "works": []
}
```

`snapshot_id` 是幂等键，不等同于数据库自增主键。服务端应建立 `(agent_id, snapshot_id, collection_mode)` 唯一约束，重复上传返回第一次处理结果，不重复写指标。

## 4. 数据库迁移设计

迁移脚本建议新增为 `docs/migrations/v2.10.2.sql`，但只有在 Phase 2/3 契约与写入代码准备完成后才允许应用。迁移必须在生产库备份、`PRAGMA integrity_check` 和磁盘余量检查通过后执行。

### 4.1 `creator_content_items`：平台中立身份主表

保留现有行和现有唯一约束，新增：

- `updated_at DATETIME`；
- `last_seen_at DATETIME`；
- 索引 `idx_creator_items_platform_item(platform, platform_item_id)`。

新链路不再向 `raw_json` 保存作品对象，只写 `'{}'` 作为旧 schema 的兼容值。后续大版本再通过 SQLite 表重建把 `raw_json` 改为可空并最终废弃。

注意：现有唯一约束为 `(account_id, platform, platform_item_id)`，无法支持只按 `(platform, platform_item_id)` 查询。当前 `EXPLAIN QUERY PLAN` 对该查询显示扫描 covering index；新增索引后必须变为 `SEARCH ... (platform=? AND platform_item_id=?)`。

### 4.2 `douyin_works`：抖音业务读模型

在现表上以只增列方式新增：

- `content_id INTEGER`，最终 `UNIQUE` 并关联 `creator_content_items(id)`；
- `video_url TEXT`；
- `current_status TEXT`；
- `latest_play_count INTEGER NOT NULL DEFAULT 0`；
- `latest_like_count INTEGER NOT NULL DEFAULT 0`；
- `latest_comment_count INTEGER NOT NULL DEFAULT 0`；
- `latest_share_count INTEGER NOT NULL DEFAULT 0`；
- `latest_collect_count INTEGER NOT NULL DEFAULT 0`；
- `last_discovered_at DATETIME`；
- `last_metrics_at DATETIME`；
- `next_metrics_at DATETIME`。

现有 `play_count` 等列暂时作为兼容镜像，读路径切换完成前双写；后续版本再废弃，不能在 v2.10.2 直接删除。

已有 `UNIQUE(account_id, aweme_id)` 和 `(account_id, publish_time DESC)` 索引满足身份及发布时间查询。增加 `(account_id, next_metrics_at)` 支持按作品年龄选择待刷新集合。

### 4.3 `douyin_work_metrics`：不可覆盖的指标事实表

```sql
CREATE TABLE douyin_work_metrics (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  work_id INTEGER NOT NULL,
  play_count INTEGER NOT NULL DEFAULT 0,
  like_count INTEGER NOT NULL DEFAULT 0,
  comment_count INTEGER NOT NULL DEFAULT 0,
  share_count INTEGER NOT NULL DEFAULT 0,
  collect_count INTEGER NOT NULL DEFAULT 0,
  interaction_rate REAL NOT NULL DEFAULT 0,
  captured_at DATETIME NOT NULL,
  snapshot_id TEXT NOT NULL,
  FOREIGN KEY(work_id) REFERENCES douyin_works(id) ON DELETE RESTRICT,
  UNIQUE(work_id, snapshot_id)
);

CREATE INDEX idx_douyin_work_metrics_time
  ON douyin_work_metrics(work_id, captured_at DESC);
```

使用 `UNIQUE(work_id, snapshot_id)` 而不是只用时间戳，避免重试产生重复事实。指标写入使用 `INSERT ... ON CONFLICT DO NOTHING`，禁止 `UPDATE` 覆盖历史。

现有 `douyin_work_snapshots` 暂不删除。迁移期先做只读回填校验，再切换读路径；它只在 Phase 5 获得数据处置批准后才能归档或下线。

### 4.4 `douyin_work_metric_daily_summary`

```sql
CREATE TABLE douyin_work_metric_daily_summary (
  work_id INTEGER NOT NULL,
  summary_date DATE NOT NULL,
  max_play_count INTEGER NOT NULL DEFAULT 0,
  like_delta INTEGER NOT NULL DEFAULT 0,
  interaction_rate REAL NOT NULL DEFAULT 0,
  sample_count INTEGER NOT NULL DEFAULT 0,
  first_captured_at DATETIME NOT NULL,
  last_captured_at DATETIME NOT NULL,
  PRIMARY KEY(work_id, summary_date),
  FOREIGN KEY(work_id) REFERENCES douyin_works(id) ON DELETE RESTRICT
);
```

先聚合、校验行数/边界值/校验和，再标记源指标为“可归档”。物理删除超过 365 天的源行属于 Phase 5，必须再次确认。

容量边界：50,000 metric/日即 18,250,000 行/年。若每个作品每天本来只有一个快照，日汇总不会降低行数；要长期稳定，还需要年度 SQLite 冷库或对象存储，或者对超过 2 年的数据进一步做周/月汇总。该决策应在 Phase 4 容量压测后确定，不能只依赖 365 天日汇总。

### 4.5 原始响应生命周期

新增 `raw_data_archive_policy` 配置表，至少包含：

- `platform`、`response_type`；
- `hot_retention_days`，默认 7；
- `archive_target`，默认 `metadata_only`；
- `enabled`、`created_at`、`updated_at`。

为 `creator_api_raw_records` 增加 `response_type`、`payload_expires_at`、`payload_purged_at`、`object_storage_key`。保留现有 `hash`、`api_url`、`created_at`。

由于 `response_json` 当前为 `NOT NULL`，7 天后清空正文需要 SQLite 表重建，不能靠简单 `ALTER COLUMN`。执行顺序必须是：新表创建 -> 批量复制 -> 行数/哈希校验 -> 原子改名。Phase 1 只设计，不执行。Agent 2.10.2 默认不上传所有网络响应，只上传白名单诊断元数据；人工 `full_snapshot` 才可按策略上传必要原文。

### 4.6 同步日志与契约审计

`douyin_sync_logs` 新增：

- `contract_version TEXT`；
- `collection_mode TEXT`；
- `snapshot_id TEXT`；
- `cleanup_at DATETIME`（计划清理时间）；
- `summary_json TEXT`（小型计数摘要，禁止响应正文）。

索引建议：

- `(account_id, sync_time DESC)`：已存在；
- `(cleanup_at)`：清理扫描；
- `(snapshot_id)`：任务追踪。

日志保留 180 天。清理任务先输出 dry-run 数量和最早/最晚时间，生产执行需独立开关。

## 5. 三种同步模式的服务端语义

| 模式 | 主表 | 最新指标 | 指标历史 | 原始响应 |
| --- | --- | --- | --- | --- |
| `discover` | 插入新作品；已有作品仅更新非空元数据和 `last_discovered_at` | 不更新 | 不写 | 默认不写正文 |
| `metrics_refresh` | 校验作品身份；必要时补齐非空元数据 | 更新 `latest_*` 和 `last_metrics_at` | 追加一条幂等事实 | 默认不写正文 |
| `full_snapshot` | 人工触发完整校验与缺失报告，不因缺失自动删除作品 | 更新 | 追加 | 仅按白名单和保留策略 |

任何模式都不得用空字符串覆盖已有标题/封面，不得用缺失指标或默认 0 覆盖已知最新指标。

## 6. 采集频率与索引化选择

服务端根据 `publish_time`、`last_metrics_at` 计算 `next_metrics_at`：

- 0–7 天：配置化高频，建议默认每日 2 次；
- 7–30 天：每日；
- 30–180 天：每 7 天；
- 180 天以上：每 30 天。

“新作品发现”每日 1–2 次，只翻页发现身份和元数据。指标刷新由服务端返回明确 ID 列表，Agent 不应为刷新指标再次全量深翻所有页面。

## 7. 查询计划验收

当前只读副本结果：

| 查询 | 当前计划 | 结论 |
| --- | --- | --- |
| `douyin_works(account_id, aweme_id)` | `SEARCH ... sqlite_autoindex_douyin_works_1` | 通过 |
| `douyin_works account + publish_time` | `SEARCH ... idx_douyin_works_account_publish` | 通过 |
| `douyin_work_snapshots(work_id, snapshot_time)` | `SEARCH ... idx_douyin_work_snapshots_time` | 通过 |
| `creator_content_items(platform, platform_item_id)` | `SCAN ... sqlite_autoindex_creator_content_items_1` | 不通过，需新增索引 |

Phase 4 必须在 100,000 作品合成库执行同一组 `EXPLAIN QUERY PLAN`，并设置可复现性能门槛：身份查询 P95 < 10 ms、账号最近 30 条作品 P95 < 50 ms、单作品 365 天指标 P95 < 50 ms（固定测试硬件、预热后至少 1,000 次）。

## 8. 分阶段实施与回滚

### Phase 1：迁移设计（本文件）

- 只读审计、设计确认、无生产数据变化。

### Phase 2：Agent 契约升级

- 大整数安全解析；端点/容器白名单；分页发现；三种模式；上传白名单；契约单测。
- Agent 保持 2.10.1 配置文件兼容。

### Phase 3：服务端 Normalizer 升级

- 新旧契约分流；统一 `DouyinWorkInput`；同事务双表写入；幂等任务收据。
- 移除运行时 `DELETE`，改为污染候选报告。

### Phase 4：指标历史表与容量验证

- 先建表和索引，再双写、校验、切读；执行 10 万作品压测和查询计划验收。

### Phase 5：历史污染与生命周期

- 只生成差异/归档/清理清单；得到明确批准后才清空 raw 正文或处置污染行。
- 删除前必须有可恢复备份、行级审计文件和回滚演练。

### Phase 6：生产灰度

- 先单账号 `discover`，再单账号 `metrics_refresh`，最后扩大比例。
- 灰度开关至少包括：新契约接收、双写、指标新表读、raw 生命周期任务。
- 任一步出现 ID 集合差异、错误率上升或查询退化，关闭开关并回到 v2.10.1 读写路径；新增表保留，不逆向删除。

每个 Phase 单独提交 Git，禁止把 schema、Agent、服务端和数据清理混入同一提交。

## 9. 测试与验收矩阵

1. 解析过滤：`12 aweme + 12 music + 7 module` 必须输出 12。
2. 长 ID：从网络响应到数据库查询全程严格等于字符串 `7663799549412758193`；number 类型契约应返回 400。
3. 分页：三个真实作品页 `12 + 12 + 7` 输出 31，按 `aweme_id` 去重。
4. 幂等：同一 `snapshot_id` 重放 10 次，作品数和指标数均不增长；十个不同指标快照只增长指标数。
5. 模式隔离：`discover` 不改变任何 `latest_*`；`metrics_refresh` 同时更新最新缓存并追加历史。
6. 双表一致：按账号比较 `creator_content_items.platform_item_id` 与 `douyin_works.aweme_id`，差集均为 0。
7. 容量：100,000 作品、按年龄分布生成指标；验证 DB 大小、写入耗时、P95 查询和 WAL 峰值。
8. 生命周期：只在副本执行 dry-run；验证 7/180/365 天边界，不触碰边界内数据。
9. 兼容：无 `contract_version` 的 v2.10.1 fixture 仍成功；2.10.2 缺少三项契约字段必须失败并有明确错误码。

## 10. 风险与待确认决策

1. **主表职责**：建议以 `creator_content_items` 为跨平台身份主表、`douyin_works` 为抖音读模型。这符合多平台扩展和现有 API 授权模型；如坚持 `douyin_works` 为唯一物理主表，需要重写通用 Creator Analytics 查询，风险更高。
2. **长 ID 解析**：只改 TypeScript 类型和 SQLite 列类型无效，必须改网络响应解析入口。
3. **历史清理授权**：7 天 raw 正文清空、365 天指标归档、180 天日志清理均是数据处置动作，Phase 5 前不执行。
4. **日汇总容量**：每天一次采样时，日汇总不会压缩 18,250,000 行/年的规模；需要确认是否接受年度冷库或更长期周/月汇总。
5. **现有删除脚本**：`v2.10.1.sql` 已包含 `DELETE`，不得在生产重复执行；后续应提供无删除的修订迁移，而不是修改已发布迁移的历史语义。
6. **SQLite 锁与磁盘**：表重建、回填和大索引创建会放大 WAL 和临时磁盘，必须离线或灰度窗口执行，并预留至少数据库体积 2–3 倍空间。
