# 上线回滚方案

本文档是第 10 轮上线前预案。本轮不执行生产部署、不重启生产服务、不修改生产配置。

## 上线前备份

上线前必须完成：

1. 备份 SQLite 数据库文件。
2. 记录当前代码 commit 或 tag。
3. 备份当前构建产物或记录可重新构建的版本。
4. 记录当前 PM2 状态：`pm2 status`。
5. 记录当前服务环境变量来源，不记录密钥明文。
6. 确认回滚负责人和验证人。

## 代码回滚

可选方案：

```bash
git reset --hard <previous_commit>
```

或切回上一稳定 tag：

```bash
git checkout <previous_tag>
```

随后执行：

```bash
npm ci
npm run build
pm2 restart xmt-api --update-env
pm2 save
```

## 数据库回滚

- 推荐使用上线前数据库备份恢复。
- 本次新增表原则上不影响旧业务。
- 如果只回滚代码，可保留新增表不使用。
- 如果必须完全回滚数据库，停止服务后用上线前备份覆盖当前数据库文件。
- 不建议在线上手动 `DROP TABLE` 删除新表，避免误删或破坏后续排查现场。

## 服务回滚

后端服务：

```bash
pm2 restart xmt-api --update-env
pm2 save
```

静态资源或反向代理：

```bash
systemctl reload caddy
```

## 验证路径

回滚后至少验证：

- `/api/health`
- 登录页
- 首页
- 日报页
- 复盘页
- 编辑器页面
- 成片制作详情页

## 静态资源缓存

- 前端重新构建后，浏览器可能仍命中旧缓存。
- 验证时使用强制刷新或无痕窗口。
- 如后续接入 CDN，需要补充 CDN 缓存刷新步骤。

## 回滚完成条件

- API 健康检查通过。
- 登录和首页可用。
- 原有核心业务页面可打开。
- PM2 状态正常。
- Caddy reload 无错误。
- 记录回滚时间、版本、数据库备份文件和验证结果。
