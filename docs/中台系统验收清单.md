# 中台系统验收清单

## 文档目的

本文档用于定义 XMT 从“项目型后台”走向“完整新媒体中台系统”的验收标准。

它不是新功能需求列表，而是后续治理和迭代时的核对清单。任何阶段性完成结论都必须能被真实代码、真实页面、真实数据库、真实运行行为或校验命令证明。

## 验收原则

1. 现状与规划必须分开描述，禁止把规划写成已完成。
2. 每个验收项必须能找到证据来源。
3. 业务主链路优先于旁支功能。
4. 平台化能力必须建立在当前单体系统稳定的基础上。
5. 文档、代码、数据库、权限和 UI 口径必须同步推进。

## P0：内容生产中台稳定验收

P0 的目标是证明当前系统作为内部内容生产中台是稳定、清晰、可维护的。

| 验收维度 | 验收标准 | 证据来源 |
|---|---|---|
| 主流程 | 选题、创作、拍摄、发布、分析主链路可串联 | 路由、页面、API、数据库记录 |
| 数据语义 | 核心表和字段含义明确，文档不与实际 schema 冲突 | `数据库设计.md`、`数据库语义冻结层.md`、`api/database/db.ts` |
| 权限边界 | 角色权限、数据归属、敏感接口守卫口径一致 | 权限治理三件套、后端路由守卫、前端入口 |
| Workflow | Workflow Engine 当前模式、阻断条件和观测能力可解释 | `系统架构.md`、`Codex开发规范.md`、workflow 相关代码 |
| 编辑器 | 编辑器初始化、写入、协作和保存路径不分裂 | 编辑器运行时规则、写入一致性规范、ContentEditor |
| UI 一致性 | 核心页面布局、状态提示、操作入口遵循统一规范 | UI 设计规范、组件规范、目标页面 |
| 文档闭环 | 修改前可从文档索引找到主文档，修改后同步 docs | `文档索引.md`、变更记录、校验结果 |
| 校验命令 | 常规修改后 `npm run check` 和 `npx tsc --noEmit` 通过 | 命令输出 |

## P1：品牌中台验收

P1 的目标是证明 XMT 不只是内部项目，而具备自有品牌中台产品的基本交付能力。

| 验收维度 | 验收标准 | 证据来源 |
|---|---|---|
| 品牌表达 | 系统名称、视觉风格、登录体验和基础品牌信息可统一 | 品牌设置页面、UI 规范、项目说明 |
| 模块边界 | 核心模块职责清晰，历史兼容结构有说明 | 系统架构、重构规划、代码目录 |
| 协作体验 | 评论、版本、通知、编辑状态和协同提示可被用户理解 | 编辑器文档、协作页面、消息模块 |
| 管理交付 | 配置、备份、导出、权限、文档具备可交付说明 | 系统设置、备份、导出、权限文档 |
| 产品一致性 | 左侧导航、页面标题、状态文案和关键操作保持中文一致 | UI 页面、组件规范 |

## P2：行业通用中台验收

P2 的目标是证明系统具备进一步抽象为行业通用平台的基础，但该阶段仍必须以真实落地为准。

| 验收维度 | 验收标准 | 证据来源 |
|---|---|---|
| 多团队预留 | 多团队、多品牌、多业务线模型有清晰规划或实现边界 | 产品路线图、系统架构、数据库设计 |
| 模板化能力 | 流程、权限、内容结构具备模板化方向 | Workflow 文档、权限矩阵、产品路线 |
| 平台配置 | 模块开关、品牌配置、流程配置的边界明确 | 产品路线图、系统设置相关代码 |
| 行业复用 | 能说明哪些能力可复用，哪些仍绑定当前场景 | 产品愿景、项目说明、验收清单 |

## 禁止作为完成证据的内容

以下内容不能单独作为“已完成”的证据：

1. 只有规划文档，没有真实代码或页面。
2. 只有页面入口，没有后端守卫或数据语义。
3. 只有后端接口，没有前端入口或使用路径。
4. 只有历史记忆，没有当前工作区证据。
5. 只有搜索不到问题，没有正向验证。
6. 只有类型检查通过，但没有覆盖对应业务要求。

## 后续使用方式

1. 功能迭代前，先通过 `文档索引.md` 找到对应主文档，再查看本清单的验收维度。
2. 功能完成后，在最终结果中说明本次改动影响了哪些验收维度。
3. 如果新增重要能力，应更新本清单、产品路线图或功能优先级中的对应口径。
4. 如果某项能力仍处于规划阶段，必须明确标注为规划，不得写成已实现。
