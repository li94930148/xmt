# 前后端权限对齐表

## 文档说明

本文档用于建立前端页面、菜单、按钮与后端接口守卫之间的映射。

原则：

1. 前端权限只负责体验，不作为安全边界。
2. 后端接口权限是最终安全边界。
3. 页面级权限不能替代操作级权限。
4. 数据归属和 workflow decision 是权限体系的一部分，不能简单等同于权限码。

## 数据归属优先模型

P1-004C 后，以下页面和接口不视为“遗漏权限”，而是明确采用 owner / scope 模型：

1. 选题管理：页面登录可进，后端按创建者、负责人和 privileged user 限制数据范围。
2. 创作管理：页面登录可进，后端按关联选题 scope 限制查看和编辑范围；`workflow:production` 代表更强管理能力，不是唯一入口条件。
3. 资源库：页面登录可进，后端按本人资源或 privileged user 限制查看、编辑、删除。
4. 排期日历：页面登录可进，后端按本人事件、关联选题或 privileged user 限制范围。

页面级权限只决定是否进入页面，不代表可访问全量数据。危险操作仍必须由后端权限点或 owner / scope 兜底。

## 前端权限处理规则

统一规则：

1. 没有页面查看权限：不显示菜单。
2. 直接访问路由但无权限：显示 `AccessDeniedState`。
3. 有页面权限但无操作权限：按钮隐藏或禁用。
4. 危险操作没有权限：不允许打开确认弹窗。
5. 后端返回 403：提示“当前账号无权限执行此操作”。
6. 后端返回 401：按现有登录失效逻辑处理。

当前已落地：

1. `RoleGuard` 支持权限码和 require all。
2. `Sidebar` 通过 `navigationSections` 的 `permissions` 控制菜单可见性。
3. `CommandPalette` 使用同一导航权限逻辑过滤页面入口。
4. `AccessDeniedState` 已作为无权限页面状态。

当前需注意：

1. 侧栏未限制的页面不代表后端没有数据范围限制。
2. 只读分析类页面前后端已统一到 `analytics:view`。
3. 设置中心是个人设置与系统设置混合页，应继续用区块级权限控制。

## 本轮处理建议

本轮不建议直接大规模改代码。建议下一阶段按以下顺序处理：

1. 后续如拆分关键状态流转，再评估是否对特定状态变更接入 `topic:status`。
2. 平台化前再补资源、日程等辅助模块专属权限。
3. 后续治理评论删除本人/删除任意评论的权限拆分。

## 当前结论

1. 用户、角色、权限管理链路前后端基本一致。
2. 导出、备份、抖音、发布、成片等敏感主路径基本一致。
3. 内容分析与协作治理页面已在 P1-004B 完成后端 `analytics:view` 收口。
4. 选题、创作、资源、日历当前采用“登录 + 数据归属”模型，P1-004C 后该模型已冻结为正式权限口径。
