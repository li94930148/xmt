# 功能优先级

## 说明

本优先级文档基于两个维度综合判断：

1. 当前项目真实已有能力与治理现状
2. “做成自有品牌中台，再扩展为行业通用系统”的长期目标

优先级分为：

1. `P0` 当前必须优先处理
2. `P1` 品牌中台阶段重点建设
3. `P2` 行业平台化阶段逐步建设

## P0 当前必须优先处理

### 1. 文档体系统一

原因：

1. 当前项目正在治理期
2. 文档是后续所有协作和重构的依据
3. 品牌化前必须先统一认知
4. 阶段完成必须能通过 `中台系统验收清单.md` 找到可验证证据

### 2. 数据与结构一致性治理

范围：

1. 数据库结构与初始化脚本口径
2. 历史模块与主路径边界
3. 页面、组件、文档描述一致性
4. 权限矩阵、前端入口、后端守卫一致性

原因：

1. 不先统一真实现状，后续品牌化会建立在不稳定基础上

当前进展：

1. P1-004A 已完成权限现状盘点、权限矩阵、前后端权限对齐表。
2. P1-004B 已完成敏感只读治理 API 权限收口。
3. P1-004C 已冻结数据归属优先模型，并完成评论创建与灵感删除的最小收口。
4. P1-004D 已完成选题创建 `topic:create` 收口，并确认状态变更继续保留 owner / scope + 状态机。

后续优先级：

1. 不再把 owner / scope 模型视为遗漏权限。
2. 优先治理关键状态流转、评论删除、资源库和日历这类尚未细分的操作权限。
3. 不在未确认产品口径前修改默认角色权限种子。

### 3. UI 与组件规范落地

范围：

1. 页面模板统一
2. 状态标签统一
3. 表格、表单、弹窗统一
4. 编辑器与普通页面边界梳理

原因：

1. 品牌中台必须先有统一产品质感

### 4. 核心链路稳定性

范围：

1. 选题
2. 创作
3. 拍摄
4. 发布
5. 分析

原因：

1. 这是品牌中台最核心的产品价值

## P1 品牌中台阶段重点建设

### 1. 品牌配置能力

建设方向：

1. 系统名称配置
2. 登录风格配置
3. 品牌视觉配置
4. 系统说明与版权信息配置

意义：

1. 从项目走向产品的第一步

### 2. 协作增强能力

建设方向：

1. 评论与批注增强
2. 编辑反馈更清晰
3. 版本状态更明确
4. 流程通知更完整

意义：

1. 为后续协同编辑平台打基础

### 3. 模板与流程配置增强

建设方向：

1. 工作流模板标准化
2. 选题模板扩展
3. 多类型业务流程支持

意义：

1. 提高品牌中台的复制能力

### 4. 管理与交付能力增强

建设方向：

1. 配置清晰
2. 文档完整
3. 结构稳定
4. 便于复制部署

意义：

1. 为内部多团队、多场景使用做准备

## P2 行业平台化阶段逐步建设

### 1. 多组织与多空间能力

建设方向：

1. 多团队
2. 多品牌
3. 多业务线
4. 多项目空间

### 2. 行业模板化能力

建设方向：

1. 传媒行业模板
2. 品牌营销模板
3. 企业内容协作模板
4. 轻项目流程模板

### 3. 平台配置能力

建设方向：

1. 模块开关
2. 流程开关
3. 权限模板
4. 品牌模板

### 4. 平台级数据与治理能力

建设方向：

1. 审计
2. 报表
3. 备份策略
4. 数据导出与归档

## 不建议当前优先投入的事项

以下方向不是不能做，而是不建议抢在当前阶段前面：

1. 大规模推翻式重构
2. 过早拆微服务
3. 脱离当前真实场景去做泛化设计
4. 在 UI、组件、数据库尚未稳定前过早扩行业功能

## 当前优先级结论

当前最正确的投入顺序是：

1. 先把现有系统做稳
2. 再把它做成有统一品牌感的中台产品
3. 最后再做行业化抽象

因此，现阶段最重要的不是“继续堆功能”，而是：

1. 统一
2. 治理
3. 沉淀
4. 可复制
5. 可验收
## Workflow Engine 当前优先级

当前仓库中的 Workflow Engine 已从 Shadow / Decision / Partial Control 继续推进到 `v2_strict` 强约束阶段。后续优先级应以“保证审批链路可控、规则口径一致、回滚路径清晰”为主，而不是继续扩展新的流程功能。

### P0：规则一致性与生产安全

1. 保持 `shared/workflow/policy.ts`、审批 API、WorkflowDesigner UI 三处规则口径一致。
2. 任何 `v2_strict` 阻断逻辑调整，都必须先说明会影响哪些状态转换、审批人判断和 UI 保存行为。
3. 不得在未确认的情况下修改 `topic.status` 行为、历史审批数据语义或工作流数据库结构。
4. Shadow / Analytics / Decision 输出必须继续作为排查和回滚依据保留。

### P1：规则治理能力增强

1. 补充 Workflow Engine v2 的异常样本、误报样本和回归用例。
2. 评估是否将 decision metadata 正式沉淀到审批记录中；当前仓库未修改 `approval_records` schema。
3. 评估 WorkflowDesigner 模板版本化，避免严格规则上线后历史模板语义不清。
4. 继续整理 WorkflowPolicy 与后端固定状态机之间的边界。

### P2：平台化流程能力

1. 在规则稳定后，再考虑可配置 workflow engine、复杂审批策略和跨业务线流程模板。
2. 暂不优先引入新的流程引擎框架。
3. 暂不优先重写现有工作流状态机 schema。
