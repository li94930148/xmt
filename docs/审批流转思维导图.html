① 选题创建
成员/编导/管理员均可创建，自动进入待审核状态
▼
用户操作
填写表单：标题、描述、大纲、平台、截止日期
可指定负责人（assignee_id）
使用 AddTopic.tsx 页面，Quill 编辑器撰写描述
→ POST /api/topics
后端处理
1. 插入 topics 表，status = pending
2. 插入 topic_history 记录（action: created）
3. 插入 activity_log 操作日志
4. 发送消息通知（创建者 + 所有管理员/编导）
5. Socket 广播 topic:created
→ 触发通知
通知规则
创建者 收到 "选题提交通知"
admin + director 收到 "新选题待审核"
🔍
② 审核阶段
管理员/编导审核选题，通过或驳回
▼
审核操作
前端：TopicDetail.tsx 审核弹窗
接口：POST /api/topics/:id/audit
权限：requireRole(['admin', 'director'])
→ 审核通过 path
✅ 审核通过
1. 状态变更：pending → approved
2. 可指派负责人（assignee_id）
3. 自动创建 production 记录（v1.0, draft）
4. 记录 topic_history（action: approved）
5. 消息通知创建者：审核通过
6. 如指派了新负责人，通知新负责人
→ 审核驳回 path
❌ 审核驳回
1. 状态变更：pending → rejected
2. 填写驳回原因（comment）
3. 记录 topic_history（action: rejected）
4. 消息通知创建者：审核驳回 + 原因
5. 创建者可修改后重新提交 → 回到 pending
✍️
③ 创作阶段
Tiptap 富文本编辑器，支持版本管理、批注、高亮
▼
编辑剧本
前端：ProductionDetail.tsx + editor/Editor.tsx
编辑器：Tiptap（HTML + JSON + Markdown 三格式存储）
功能：标题/格式/表格/图片/任务列表/高亮/批注/目录
→ 保存 / 提交审核
版本管理
接口：PUT /api/workflow/production/:id
1. 旧版本存入 production_history
2. 版本号自增：minor → v1.1 / major → v2.0
3. 状态变更：draft → review → approved
→ 创作审核通过
自动流转
当 production.status = approved 时：
1. 选题状态：approved → production → shooting
2. 自动创建 shooting 记录（如不存在）
3. 消息通知："创作已通过审核，进入成片制作环节"
🎬
④ 拍摄/成片制作
拍摄计划、场地设备、剧本副本
▼
拍摄管理
前端：ShootingDetail.tsx
接口：PUT /api/workflow/shooting/:id
权限：requireRole(['admin', 'director'])
字段：拍摄日期、场地、设备、状态、剧本副本
→ 标记完成
完成流转
当 shooting.status = completed 时：
1. 选题状态：shooting → publishing
2. 自动创建 publishing 记录（如不存在）
3. 剧本副本（script_content）传递到发布管理
4. 消息通知："成片制作已完成，进入发布管理环节"
📤
⑤ 发布阶段
平台发布、URL 记录、数据指标
▼
发布管理
前端：PublishingDetail.tsx
接口：PUT /api/workflow/publishing/:id
权限：requireRole(['admin', 'director'])
字段：平台、发布 URL、发布时间、播放/点赞/分享/评论数
→ 标记已发布
完成流转
当 publishing.status = published 时：
1. 选题状态：publishing → completed
2. 自动创建/更新 analytics 记录
3. 消息通知："发布完成"
4. 选题进入终态 completed，生命周期结束
⚙️
⑥ 审批流模板引擎
可自定义审批节点、审批人、流程顺序
▼
模板结构
workflow_templates — 审批流模板
workflow_nodes — 审批节点（有序）
approval_records — 审批记录
前端：WorkflowDesigner.tsx 可视化设计
→ 节点配置
节点属性
name 节点名称（如"内容审核"）
node_order 执行顺序（1,2,3...）
status_from → status_to 状态转换
approver_type 审批人类型： role / user / creator
approver_value 审批人值（角色名/用户ID）
is_required 是否必须审批
→ 默认模板
标准选题流程（5 个节点）
① 内容审核 pending→approved role:director
② 创作阶段 approved→production role:member
③ 拍摄阶段 production→shooting role:director
④ 发布阶段 shooting→publishing role:director
⑤ 完成确认 publishing→completed role:director

⑦ 权限控制
RBAC 模型：角色 → 权限 → 操作
▼
角色权限矩阵
操作admindirectormember
创建选题✅✅✅
审核选题✅✅❌
删除选题✅✅❌
编辑剧本✅✅✅ (仅自己的)
拍摄管理✅✅❌
发布管理✅✅❌
状态推进✅✅✅ (受限)
系统设置✅❌❌
→ 中间件链
权限检查流程
authenticate → JWT 验证 → 注入 req.user
requireRole → 角色白名单校验
requirePermission → 细粒度权限校验（带 5 分钟缓存）
admin 角色自动跳过所有权限检查

⑧ 通知系统
站内消息 + Socket 实时推送 + 桌面通知
▼
审批流触发的通知
事件接收人类型

选题创建admin + directorwarning

审核驳回创建者error
任务指派被指派人info
创作审核通过操作者success
成片完成操作者success
发布完成操作者success
状态变更创建者 + 负责人info
→ 推送渠道
三通道推送
站内消息 → messages 表 → 消息中心页面
Socket.IO → 实时推送 → 弹窗 Toast
桌面通知 → 浏览器 Notification API（需 HTTPS）
