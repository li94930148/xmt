# 数据复盘系统后端设计

## 范围

数据复盘系统后端定位为“结论 + 指标快照 + 行动项”的业务闭环。第 8 轮仅在既有表结构上增强行动项、日报风险联动、导出和消息提醒，不新增表字段，不修改日报表结构，不删除 `analytics` / `weekly-report` 能力。

## 数据模型

已存在表：

- `retro_templates`：复盘模板。
- `retrospectives`：复盘主表，保存范围、周期、状态、结论和版本。
- `retro_metric_snapshots`：复盘指标快照，生成后固化。
- `retro_actions`：复盘行动项。
- `retro_audit_logs`：复盘审计日志。

第 8 轮不涉及 schema 变更。

## 权限

后端继续使用既有权限码：

- `analytics:retro:view`
- `analytics:retro:create`
- `analytics:retro:publish`
- `analytics:retro:archive`
- `analytics:retro:action_manage`
- `analytics:retro:template_manage`

行动项规则：

- `canManageActions` 用户可创建和完整维护行动项。
- 行动项负责人可更新自己的状态和结果。
- 归档复盘不允许新增行动项，但已有行动项负责人仍可更新进展和结果。

## API

基础路径：`/api/retrospectives`

- `GET /templates`：获取模板。
- `POST /templates`：创建模板。
- `GET /`：查询复盘列表。
- `POST /`：创建复盘。
- `GET /actions/mine?status=open|todo|doing|done|cancelled|all`：获取我的复盘行动项。
- `PATCH /actions/:actionId`：更新行动项。
- `GET /:id`：获取复盘详情。
- `PUT /:id`：更新草稿复盘。
- `POST /:id/snapshot`：生成或替换指标快照。
- `POST /:id/publish`：发布复盘。
- `POST /:id/archive`：归档复盘。
- `POST /:id/actions`：创建行动项。
- `GET /:id/daily-risks`：读取复盘周期内日报风险/阻塞分段。
- `GET /:id/export?type=markdown|html`：导出复盘内容。

路由顺序要求：`/actions/mine`、`/:id/daily-risks`、`/:id/export` 必须注册在 `/:id` 之前。

## 日报风险联动

- 后端仅读取 `daily_reports` 和 `daily_report_items`。
- 查询范围来自复盘 `period_start` 到 `period_end`。
- 当前识别分段：`risk`、`risks`、`blocker`、`blockers`。
- 接口返回风险来源，不创建行动项；创建行动项仍走 `POST /:id/actions`。
- 不更新日报内容、日报状态或日报表结构。

## 导出

- Markdown 导出返回复盘标题、类型、周期、状态、负责人、创建人、发布时间、归档时间、结论、指标快照、行动项。
- HTML 导出为可打印页面，后端会转义文本内容。
- 不生成 PDF。
- 导出接口复用详情权限，只有可查看复盘的人可以导出。

## 消息提醒

复用现有 `messages` 表和 `createMessage`：

- 复盘发布：提醒负责人、创建人、管理员/总监。
- 复盘归档：提醒负责人、创建人、管理员/总监。
- 行动项分配：提醒行动项负责人。
- 行动项更新：提醒复盘负责人、复盘创建人、行动项创建人。

提醒写入为非阻塞流程，消息失败不影响主业务操作。

## 已知限制

- 逾期提醒仅完成状态判断和设计文档，未接入定时任务。
- 没有外部平台真实数据接入，也没有模拟外部平台指标。
- 真实 token 全链路回归依赖本地可用账号或用户提供登录态。
