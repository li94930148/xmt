## 当前数据库设计特征

### 已实现

从实际数据库读取结果可以确认以下特征：

1. 数据库以内容生产流程为中心
- `topics`、`production`、`shooting`、`publishing`、`analytics` 构成主链路

2. 数据库已经扩展到协作和治理层
- 存在消息、通知、角色权限、工作流、公告、灵感、资源、成就、番茄钟、日历等结构

3. 物理外键数量有限
- 当前仅少数表使用真实外键约束
- 大量业务关联仍以逻辑关联字段为主

4. 迁移方式偏增量追加
- 从建表 SQL 可以看到多个表通过追加字段持续演进

## 真实外键关系

### 已实现

当前实际库中可确认的外键关系包括：

1. `douyin_snapshots.account_id -> douyin_accounts.id`
2. `douyin_videos.snapshot_id -> douyin_snapshots.id`
3. `notification_preferences.user_id -> users.id`
4. `role_permissions.role_id -> roles.id`
5. `role_permissions.permission_id -> permissions.id`
6. `user_roles.user_id -> users.id`
7. `user_roles.role_id -> roles.id`
8. `workflow_nodes.template_id -> workflow_templates.id`

### 说明

虽然很多表中存在 `topic_id`、`creator_id`、`operator_id`、`uploader_id`、`approver_id` 等字段，但在当前实际数据库中，这些字段大多没有被声明为物理外键。

这意味着：

1. 当前系统大量依赖应用层维护数据一致性
2. 删除和更新联动逻辑更多由业务代码负责
3. 后续数据库治理仍有较大提升空间
