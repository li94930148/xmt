# 数据库设计

## 文档说明

本文档基于以下两类真实来源整理：

1. 当前项目实际数据库文件 `data/xmt.db`
2. 当前数据库初始化与运行时迁移脚本 `api/database/db.ts`

本文档目的不是虚构理想模型，而是明确：

1. 当前真实已落地的数据结构
2. 当前仍处于兼容或过渡状态的结构
3. 后续品牌化与平台化需要重点治理的方向

## 数据库总览

当前数据库已覆盖以下业务域：

1. 用户与权限
2. 内容生产主流程
3. 协作与消息
4. 工作流与模板
5. 运营与辅助模块
6. 抖音数据
7. 系统元数据
8. 历史兼容结构

当前数据库已经具备中台型产品雏形，不再只是单表驱动的小型业务库。

## 当前兼容或过渡结构

### `editing`

当前数据库和初始化脚本中都存在 `editing` 表。

当前判断：

1. 该表属于历史兼容结构
2. 仍需保留以兼容现有数据
3. 但它不是当前前端主路径的核心模块

后续建议：

1. 核查实际使用链路
2. 明确是保留兼容、迁移替代，还是归档停用

## 真实结构与治理重点

### 已确认的特点

1. 数据库以主流程链路为中心扩展
2. 权限、协作、运营模块已进入正式结构
3. 部分业务关系仍依赖应用层维护
4. 迁移方式当前仍偏运行时增量修补

### 当前需要特别关注的问题

1. 数据库初始化脚本与长期运行数据库需要持续核对
2. 历史表、兼容表、主路径表需要明确边界
3. 平台化前必须建立更稳定的迁移管理方式

## 品牌中台阶段的数据库方向

在“自有品牌中台”阶段，数据库需要重点支持：

1. 系统品牌配置
2. 多团队/多业务线预留
3. 更稳定的模板能力
4. 更清晰的工作流与通知联动

这意味着数据库未来会更强调：

1. 配置化
2. 可演进
3. 可复制
4. 可治理

## 行业通用平台阶段的数据库方向

未来如果向行业通用平台扩展，数据库应逐步具备以下能力：

1. 多组织模型
2. 多空间/多项目模型
3. 行业模板模型
4. 系统配置与品牌配置模型
5. 更稳定的统计与审计模型

## 后续治理建议

### 已明确建议

1. 建立正式迁移机制，不再只依赖运行时补字段
2. 核查 `editing` 等历史兼容结构的去留策略
3. 持续保持数据库文档与真实结构同步
4. 为品牌化和平台化能力预留统一配置模型

### 当前结论

当前数据库已经具备做成品牌中台的基础，但距离做成可复制的行业通用系统，还需要先完成：

1. 结构稳定化
2. 迁移规范化
3. 历史结构治理
4. 配置能力模型化
