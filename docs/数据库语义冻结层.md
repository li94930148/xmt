# 数据库语义冻结层

本文档用于冻结当前数据库语义边界。本阶段不修改数据库结构，不新增迁移，不改变字段含义。

## 真实源字段

当前系统真实内容源字段为：

1. `production.content`
2. `topics.outline`
3. `topics.description`
4. `shooting.script_content`
5. `publishing.script_content`

其中：

1. `production.content` 是唯一创作正文源。
2. `topics.outline` 是选题大纲源。
3. `topics.description` 是选题描述源。
4. `script_content` 是 Shooting / Publishing 各阶段的本地内容源。

## 过渡字段

以下 Markdown / JSON 字段全部为过渡字段，不得作为可靠 canonical：

1. `content_markdown`
2. `content_json`
3. `outline_markdown`
4. `outline_json`

禁止在未完成专项迁移前，将这些字段作为主读写来源、审核事实来源或版本事实来源。

## Workflow 观测表

`workflow_shadow_logs` 只增不改。

1. 用于记录 shadow、analytics、decision 和 strict gate 相关观测信息。
2. 不做 event sourcing。
3. 不作为业务状态回放事实源。
4. 不替代 `approval_records`。
5. 不在本阶段变更 schema。

## 审批记录表

`approval_records` 是审批业务事实。

1. 当前不做 policy 字段结构化改造。
2. 当前不新增 `policy_checked`、`policy_mode`、`transition_allowed`、`approve_allowed` 等字段。
3. 如后续要结构化记录 policy metadata，必须单独评估 schema、历史数据和回滚策略。

## 系统模型

当前系统仍为状态驱动模型，不是 event sourcing 系统。

1. `topic.status` 仍是业务状态判断的重要来源。
2. workflow templates 与 approval runtime 仍由现有业务表和接口语义决定。
3. `workflow_shadow_logs` 只提供观测和排查能力。
4. Strict Enforcement 是执行前 gate，不是数据库事件溯源引擎。
