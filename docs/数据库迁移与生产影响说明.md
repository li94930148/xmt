## 默认种子

- 日报默认模板：通过种子记录和默认模板存在性检查写入。
- 复盘默认模板：通过种子记录和默认模板存在性检查写入。
- 权限种子：使用 `INSERT OR IGNORE`，重复执行不会重复写入。
- 角色权限种子：使用 `INSERT OR IGNORE`，重复执行不会重复绑定。

## 幂等性说明

- 新增表使用 `CREATE TABLE IF NOT EXISTS`。
- 新增索引使用 `CREATE INDEX IF NOT EXISTS`。
- 权限和角色权限使用 `INSERT OR IGNORE`。
- 默认模板写入前检查种子记录和现有默认模板。
- 本地重复执行 `initDatabase` 成功。

## 破坏性 SQL 检查

- 未发现针对本轮新增日报表或复盘表的 `DROP TABLE`。
- 未发现清空新增表的迁移 SQL。
- 未发现上线初始化阶段覆盖旧数据的 SQL。
- 代码中存在业务运行时 `DELETE FROM`，用于草稿条目替换、复盘快照重生成或既有模块 CRUD，不属于生产迁移破坏性 SQL。
- `api/database/db.ts` 中存在既有 `ALTER TABLE` 兼容逻辑，均为 `try/catch` 包裹的旧表字段补齐，不属于本轮新增破坏性变更。

## 是否影响旧数据

本轮新增日报和复盘表原则上不影响旧业务表。新增权限和默认模板为增量数据。只回滚代码时，可以保留新增表和种子数据不用。

## 生产上线前备份

生产上线前必须备份 SQLite 数据库。建议同时记录：

- 数据库文件路径。
- 备份文件路径和时间。
- 当前代码 commit。
- 当前 PM2 状态。
- 当前服务环境变量来源，不记录密钥明文。

## 生产迁移步骤

1. 停止写入或选择低峰窗口。
2. 备份生产 SQLite 数据库。
3. 拉取代码并安装依赖。
4. 执行 `npm run check` 和 `npm run build`。
5. 执行数据库初始化入口，让建表、索引和种子幂等执行。
6. 重启 API 服务。
7. 验证 `/api/health`、登录、日报、复盘、编辑器和成片制作详情页。
8. 保留备份直到完整验收通过。

## 回滚建议

- 优先代码回滚到上一稳定 commit 或 tag。
- 如果只回滚代码，可保留本次新增表和种子数据，不会被旧代码使用。
- 如果必须完全回滚数据库，使用上线前数据库备份覆盖恢复。
- 不建议在线上手工 `DROP TABLE` 删除本次新增表。
