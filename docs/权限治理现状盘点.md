## 当前角色清单

| 角色 | 中文名 | 当前定位 | 当前实际权限 | 前端可见范围 | 后端访问范围 | 风险判断 |
|---|---|---|---|---|---|---|
| `admin` | 管理员 | 系统最高权限 | 种子中拥有全部权限；前端 `usePermission` 直接返回全权限 | 全部页面 | `requirePermission` 自动放行；仍受登录、资源存在性、workflow strict gate 约束 | 权限最大，符合系统管理员定位 |
| `director` | 编导 | 业务负责人 | 拥有非 system 模块权限，另含 `system:announcement`、`system:template`、`system:douyin` | 可进入内容流程、数据、成片、发布、审批流设计、抖音等页面 | 多数业务数据按 privileged user 放开；无用户/角色/权限/备份/设置管理权限 | 权限偏业务管理，基本合理 |
| `editor` | 编辑 | 内容编辑人员 | `topic:view`、`topic:update`、`workflow:production`、`workflow:comment` | 默认可进入未加权限的选题、创作、资源、日历、灵感等页面；不可进入成片、发布、数据、系统管理 | 可访问本人创建或被指派的内容；部分仅登录接口可访问 | 创作相关能力清晰，但页面级开放范围比权限点更宽 |
| `member` | 成员 | 普通协作成员 | `topic:create`、`topic:view`、`topic:update`、`workflow:comment` | 默认可进入未加权限的选题、创作、资源、日历、灵感等页面 | 可访问本人创建或被指派内容；可创建选题、评论 | 与 editor 边界较近，需要后续细化 |

说明：

1. 当前不存在 `viewer`、`guest` 等真实落地角色。
2. `requireRole` 中间件仍存在，但路由层未发现实际使用，当前主路径是 `requirePermission` + 资源归属判断 + workflow decision。
3. `src/constants/index.ts` 的 `ROLE_MAP` 已补齐 `editor`，与 `src/lib/roles.ts` 和真实角色清单保持一致。

## 数据归属优先模型

P1-004C 后，选题、创作、资源库、排期日历等协作型业务模块明确采用“权限码守卫 + 数据归属守卫 + Workflow strict gate 并存”的模型，不强行把所有接口改成纯权限码控制。

### privileged user

当前 privileged user 为：

1. `admin`
2. `director`

含义：

1. 可以查看大部分业务数据。
2. 可以跨用户处理内容流程。
3. 仍不能自动拥有系统管理权限，例如用户管理、角色权限、备份、系统设置仍需对应权限点。
4. 仍受 Workflow strict gate 约束。
5. 仍受删除、审核、导出等危险操作权限点约束。

### owner / scope user

普通用户的数据范围来自：

1. 本人创建。
2. 被指派。
3. 关联选题。
4. 参与协作。
5. 本人消息 / 本人配置。
6. 业务接口内明确允许的协作范围。

### 页面级权限不等于数据全量权限

1. `editor` / `member` 可以进入选题页面，但只能看到自己相关的数据。
2. `member` 可以进入创作页面，但不代表拥有全部创作管理权限。
3. 资源库和日历可以开放入口，但具体数据由 owner / scope 控制。
4. `admin` / `director` 可看更多业务数据，但系统设置、用户管理、备份仍需对应权限点。

3. `editor` 默认没有 `topic:create`，本轮不修改默认角色权限种子；`member` 默认拥有 `topic:create`，不受影响。
4. `POST /api/topics/:id/status` 暂不接入 `topic:status`，继续保留 `canAccessTopic + 状态机`，避免阻断普通协作者推进自己相关任务。

### P2：后续优化

1. `member` 与 `editor` 的内容可见范围、编辑范围边界较近，需要结合任务分配模型细化。
2. 评论删除仍只有删除任意评论的粗粒度模型，建议后续拆分删除本人评论与删除任意评论。
3. 资源、日程缺少明确权限命名模型，平台化前再拆 `resource:*`、`calendar:*`。
4. 目前权限缓存 TTL 为 5 分钟，角色调整后需要确保管理操作调用 `clearPermissionCache` 或通过重新登录刷新。

## 当前结论

1. 系统已经具备角色、权限、角色权限映射、用户角色映射和前端权限 Hook。
2. 后端最终权限来源已初步成立，但并非所有接口都使用权限码。
3. 当前真实模型是：权限码守卫 + 资源归属守卫 + 工作流审批守卫并存。
4. 前端页面权限整体比过去更清晰，P1-004B 已完成敏感只读治理 API 的第一轮后端收口。
5. P1-004C 后，“数据归属优先”不再视为遗漏权限；后续只对危险操作、平台化权限和细粒度删除能力继续治理。
