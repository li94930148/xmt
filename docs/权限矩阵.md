# 权限矩阵

## 文档说明

本文档记录当前 XMT 真实权限矩阵，并区分“当前已落地”和“后续建议”。

当前权限来源为 `api/database/db.ts` 的默认种子：

1. `admin`：拥有全部权限。
2. `director`：拥有所有非 `system` 模块权限，另拥有 `system:announcement`、`system:template`、`system:douyin`。
3. `editor`：拥有 `topic:view`、`topic:update`、`workflow:production`、`workflow:comment`。
4. `member`：拥有 `topic:create`、`topic:view`、`topic:update`、`workflow:comment`。

## 角色定位

### admin

系统管理员。

负责：

1. 用户管理。
2. 角色权限管理。
3. 系统设置与备份。
4. 全部内容流程管理。
5. 工作流模板管理。
6. 审计与日志查看。
7. 数据导出。

当前实现中，`admin` 在前端和后端权限检查里都被视为全权限用户。

### director

业务负责人 / 内容负责人。

负责：

1. 内容流程管理。
2. 选题审核。
3. 创作协同。
4. 成片制作推进。
5. 发布管理。
6. 工作流模板配置。
7. 数据看板查看。
8. 抖音数据管理。

当前不拥有：

1. 用户管理。
2. 角色权限管理。
3. 系统备份。
4. 系统设置。
5. 成就系统管理。

### editor

内容编辑 / 执行人员。

负责：

1. 查看与编辑相关选题。
2. 参与创作阶段内容编辑。
3. 参与评论协作。
4. 查看被分配或本人创建的内容。

当前不拥有：

1. 成片制作管理。
2. 发布管理。
3. 选题创建权限。
4. 选题审核。
5. 删除类权限。
6. 数据看板和导出。
7. 系统管理。

### member

普通成员 / 协作成员。

负责：

1. 创建选题。
2. 查看与编辑相关选题。
3. 参与评论协作。
4. 处理本人或被分配任务。

当前不拥有：

1. 创作阶段专属权限点 `workflow:production`。
2. 成片制作管理。
3. 发布管理。
4. 审核和删除。
5. 数据看板和导出。
6. 系统管理。

## 数据归属优先模型

P1-004C 后，以下模块明确保留数据归属优先，不强行改成“有页面权限即可访问全量数据”：

1. 选题管理：列表、详情、编辑、状态变更继续叠加 `creator_id / assignee_id` 范围，`admin / director` 可查看更大范围。
2. 创作管理：页面登录可进入；查看、编辑继续通过关联选题范围控制。
3. 资源库：登录可进入；普通用户管理本人上传资源，`admin / director` 可调度更多资源。
4. 排期日历：登录可进入；普通用户访问本人事件或关联选题事件，`admin / director` 可查看整体排期。

`workflow:production` 的语义是更强的创作阶段管理能力，不是进入创作页的唯一条件。普通协作者仍可基于 owner / scope 访问相关创作任务。

`topic:create` 已在 P1-004D 接入创建接口和前端入口。`topic:view/update/status` 仍需叠加 owner / scope 或状态机理解，不能解释为全量数据权限。

## 页面级权限与操作级权限分离

必须保持以下规则：

1. 页面级权限只决定能否进入页面。
2. 操作级权限决定能否新增、编辑、删除、审批、导出、备份、管理配置。
3. 数据归属权限决定普通用户能看到或操作哪些具体记录。
4. Workflow decision 决定审批节点是否合法。

示例：

1. `editor` 可以进入选题管理，但不能审核或删除选题。
2. `member` 可以创建选题，但不能进入成片制作或发布管理。
3. `director` 可以进入数据复盘和导出，但不能管理用户、角色、权限、备份、系统设置。
4. `admin` 拥有全部权限，但审批仍需满足工作流 strict gate 的业务规则。

## 后端权限命名现状与建议

### 当前命名风格

当前已落地权限使用 `模块:动作`：

1. `topic:create`
2. `workflow:shooting`
3. `analytics:view`
4. `system:template`
5. `user:logs`

该风格应继续沿用。

### 优先复用现有权限

后续治理时应优先复用：

1. 数据页：`analytics:view`
2. 导出：`export:data`
3. 成片：`workflow:shooting`
4. 发布：`workflow:publishing`
5. 创作：`workflow:production`
6. 审批流模板：`system:template`
7. 审计日志：`user:logs`

### 暂不建议立即新增的权限

以下权限点可以作为后续治理候选，不建议本轮直接新增：

1. `resource:view`
2. `resource:create`
3. `resource:update`
4. `resource:delete`
5. `calendar:view`
6. `calendar:manage`
7. `workflow:audit_view`
8. `comment:delete_own`
9. `comment:delete_any`
10. `collaboration:view`

新增前应先确认产品边界、角色默认分配和兼容策略。

## 当前矩阵结论

1. 当前四类角色真实存在且可继续作为权限治理基础。
2. `admin` 与 `director` 的边界较清楚。
3. `editor` 与 `member` 的差异主要在 `topic:create` 与 `workflow:production`：`member` 可提报选题，`editor` 默认不提报选题但具备创作阶段能力。
4. 页面级权限和操作级权限已经部分分离，但仍有资源、日历、创作、选题等模块依赖数据归属而非权限码。
5. 下一阶段应优先对齐“前端已限制但后端只登录”的分析类和审计类接口。
