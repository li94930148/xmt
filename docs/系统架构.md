# 系统架构

## 架构概览

XMT 当前采用前后端同仓的单体式全栈架构：

- 前端位于 `src/`
- 后端位于 `api/`
- 共享类型位于 `shared/`
- 运行数据库位于 `data/xmt.db`
- 前端构建产物位于 `dist/`

系统以 Express 提供 API 和静态资源，以 Socket.IO 提供实时通信，以 SQLite 作为本地持久化数据库，适合当前团队内部部署和快速迭代场景。

## 当前项目架构说明

基于当前代码结构，可以归纳出以下现状：

1. 项目是典型的前后端同仓单体应用。
2. 前端使用 React SPA，通过 React Router 管理页面路由。
3. 后端使用 Express 提供 REST 风格接口，并同时托管构建后的前端静态文件。
4. 实时能力通过 Socket.IO 建立在同一服务进程中。
5. 数据库采用本地 SQLite 文件，初始化、建表、索引和增量迁移集中在 `api/database/db.ts`。
6. 当前架构强调开发效率和部署简单性，适合中小团队快速推进业务。

## 前端架构

### 技术组成

- React 18
- TypeScript
- Vite
- React Router
- Zustand
- TanStack React Query
- Tiptap
- Tailwind CSS

### 结构划分

前端主要目录职责如下：

- `src/pages/` 承载各业务页面
- `src/components/` 承载通用组件与编辑器组件
- `src/api/` 承载前端 API 调用封装
- `src/store/` 承载全局状态管理
- `src/hooks/` 承载复用逻辑，如权限、Socket、实时同步、通知等
- `src/utils/` 承载工具函数

### 运行方式

- `src/main.tsx` 为前端入口
- `src/App.tsx` 组织路由、懒加载、登录保护和整体页面结构
- 页面采用按路由懒加载方式组织，降低初始包体积
- 认证信息和部分界面配置使用本地存储持久化

### 当前特点

- 页面已经覆盖主要业务模块
- 组件和页面分层基本清晰
- 状态管理较轻量，适合当前规模
- 仍存在按业务进一步模块化拆分的空间

## 数据库架构

### 当前选型

当前数据库为 SQLite，本地数据库文件位于 `data/xmt.db`，通过 `@libsql/client` 访问。

### 初始化方式

数据库结构集中在 `api/database/db.ts` 中完成：

- 建表
- 补字段
- 初始化默认角色与权限
- 初始化默认工作流模板
- 初始化默认用户
- 执行运行时迁移
- 创建索引

### 当前主要数据域

从代码可确认的主要表包括：

- 用户与认证：`users`
- 角色权限：`roles`、`permissions`、`role_permissions`、`user_roles`
- 选题主流程：`topics`、`topic_history`
- 创作：`production`、`production_history`
- 拍摄：`shooting`
- 发布：`publishing`
- 数据分析：`analytics`
- 消息通知：`messages`、`notification_preferences`
- 协作补充：`comments`、`activity_log`、`announcements`
- 灵感池：`inspirations`、`inspiration_votes`、`inspiration_comments`
- 模板与工作流：`topic_templates`、`workflow_templates`、`workflow_nodes`、`approval_records`
- 辅助模块：`resources`、`achievements`、`user_achievements`、`pomodoro_sessions`、`calendar_events`

### 当前特点

- 数据模型已经覆盖主要业务场景
- 迁移方式偏运行时增量修补，适合快速开发
- 已建立基础索引，具备一定性能意识
- 长期建议引入正式迁移机制、外键治理和数据分层策略

## 实时通信架构

### 当前实现

系统通过 Socket.IO 提供实时能力，服务端和 API 同进程运行。

服务端侧：

- 在 `api/app.ts` 中创建 Socket.IO Server
- 在连接阶段通过 JWT 校验用户身份
- 按用户和角色加入不同房间
- 支持公共房间加入与离开

前端侧：

- 在 `src/hooks/useSocket.ts` 中维护全局单例连接
- 登录后通过 token 建立连接
- 接收新消息与未读计数更新事件
- 触发桌面通知与本地状态更新

### 当前用途

- 站内消息提醒
- 未读数同步
- 部分业务事件广播
- 管理角色房间通知

### 当前特点

- 实现简单直接，足以覆盖当前实时场景
- 适合同机部署或局域网使用
- 后续若协同编辑增强，需要引入更细的事件模型和冲突控制策略

## 部署架构

### 当前部署方式

当前仓库已提供多种部署要素：

1. 本地开发
- `npm run dev` 同时启动前端开发服务器和后端服务

2. 生产运行
- 前端先构建为 `dist/`
- 后端以 Express 提供 API 和静态资源服务

3. Docker 部署
- 已提供 `Dockerfile`
- 已提供 `docker-compose.yml`
- 支持数据卷持久化数据库和备份目录

4. 进程托管
- 已提供 `ecosystem.config.cjs`，可用于 PM2 托管

### 当前部署特点

- 部署链路较轻
- 适合单机、内网或小规模服务器场景
- 支持可选 HTTPS 证书目录
- 前后端同服务托管，维护成本较低

## 后续升级方向

### 1. 后端分层升级

- 将复杂业务逻辑从路由层下沉到服务层
- 引入更明确的仓储层和领域边界
- 让接口层更专注参数校验和响应组织

### 2. 数据库治理升级

- 引入正式迁移工具
- 完善外键和数据一致性策略
- 为核心报表和查询场景优化索引

### 3. 协同能力升级

- 扩大实时事件覆盖面
- 引入更细粒度的协作状态同步
- 为多人编辑场景准备冲突处理机制

### 4. 前端模块化升级

- 按业务域重组模块
- 减少页面层直接承载过多逻辑
- 统一 API、状态和页面的组织边界

### 5. 平台化与运维升级

- 增加日志、监控和告警能力
- 细化环境配置管理
- 为后续服务拆分和多实例部署预留空间
