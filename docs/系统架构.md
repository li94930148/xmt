# 系统架构

## 架构概览

XMT 当前采用前后端同仓的单体全栈架构：

1. 前端位于 `src/`
2. 后端位于 `api/`
3. 共享类型位于 `shared/`
4. 运行数据库位于 `data/xmt.db`
5. 前端构建产物位于 `dist/`

系统当前目标是支撑内容生产中台，同时为后续品牌化和平台化扩展保留演进空间。

## 当前项目架构说明

基于当前真实代码，可确认如下现状：

1. 项目是典型的前后端同仓单体应用
2. 前端是 React SPA，通过 React Router 管理页面
3. 后端使用 Express 提供 API、静态资源和 Socket.IO
4. SQLite 作为当前默认本地持久化数据库
5. 核心业务已经覆盖内容生产全流程和多个辅助治理模块
6. 架构更偏“快速业务沉淀 + 持续治理演进”，而不是一开始就过度服务化

## 前端架构

### 技术组成

1. React 18
2. TypeScript
3. Vite
4. React Router
5. Zustand
6. Tailwind CSS
7. Tiptap

### 当前结构

前端主要目录职责如下：

1. `src/pages/`
   - 业务页面入口
2. `src/components/`
   - 通用组件、布局组件、编辑器组件
3. `src/api/`
   - 前端接口封装
4. `src/store/`
   - 全局状态
5. `src/hooks/`
   - 复用逻辑
6. `src/utils/`
   - 工具函数
7. `src/constants/`
   - 状态与配置常量

### 当前特征

1. 路由模块已经比较完整
2. 页面数量较多，覆盖内容生产、治理、运营和系统能力
3. 编辑器能力已经形成独立能力域
4. 但列表页、详情页、弹窗和状态展示仍有重复实现

## 后端架构

### 技术组成

1. Express
2. TypeScript
3. Socket.IO
4. JWT
5. bcrypt

### 当前结构

后端主要目录职责如下：

1. `api/server.ts`
   - 服务启动入口
2. `api/app.ts`
   - 应用装配、CORS、Socket、静态资源托管
3. `api/routes/`
   - 业务路由
4. `api/middleware/`
   - 认证、权限、限流等中间件
5. `api/database/`
   - 数据库初始化与迁移
6. `api/utils/`
   - 各类辅助工具

### 当前特征

1. 后端仍以单体路由驱动为主
2. 权限和角色体系已经具备产品化基础
3. 通知、工作流、导出、备份等能力已进入后端主路径
4. 后续需要逐步增强服务层，降低路由层复杂度

## 数据库架构

### 当前选型

当前数据库为 SQLite，本地数据库文件为 `data/xmt.db`，通过 `@libsql/client` 访问。

### 当前结构特征

当前数据库已经不只是简单业务表，而是覆盖了以下几类域：

1. 用户与权限域
2. 内容生产主流程域
3. 协作与通知域
4. 工作流与模板域
5. 运营与辅助模块域
6. 抖音数据域
7. 系统元数据域

### 当前判断

1. 数据库已经具备中台型产品雏形
2. 但初始化脚本、运行时迁移、历史表兼容需要持续治理
3. 后续平台化前，必须先稳定数据结构演进方式

## 实时通信架构

### 当前实现

系统通过 Socket.IO 提供实时能力，服务端与 API 同进程运行。

### 当前用途

1. 站内消息提醒
2. 未读数同步
3. 局部业务广播
4. 管理角色房间通知

### 当前判断

当前实时通信适合内容中台场景，但如果未来升级为协同编辑平台，还需要：

1. 更细粒度事件模型
2. 更明确的协作状态同步
3. 更稳定的冲突处理策略

## 部署架构

### 当前部署方式

项目当前已具备以下部署方式：

1. 本地开发
2. 构建后单体运行
3. Docker / Docker Compose
4. PM2 进程托管

### 当前特点

1. 部署链路较轻
2. 适合内部环境、单机或轻量服务器运行
3. 适合当前品牌化产品早期快速迭代

## 后续升级方向

### 第一阶段：品牌中台架构升级

1. 稳定数据库迁移机制
2. 统一系统配置与品牌配置
3. 统一页面模板和组件体系
4. 明确主路径模块与历史兼容模块

### 第二阶段：协同编辑平台升级

1. 强化编辑器域能力
2. 强化实时协作事件模型
3. 增加服务层与业务边界
4. 为多团队协作预留更清晰的数据模型

### 第三阶段：行业通用平台升级

1. 品牌化配置能力标准化
2. 工作流模板能力标准化
3. 权限模板能力标准化
4. 多组织与多空间模型升级
5. 支持行业级模块裁剪与组合

## 架构演进原则

1. 先文档化
2. 先治理一致性
3. 再做局部重构
4. 最后才考虑更大规模的架构拆分
