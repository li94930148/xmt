# 组件规范

## 文档说明

本规范用于明确当前组件分层、基础组件职责边界和后续推广原则，避免在页面统一过程中把业务逻辑误塞进公共组件。

P1-001 页面基础框架统一阶段已正式结束。当前组件规范以“页面外层统一优先，业务内核保守处理”为总原则。

## 1. `src/components/common/` 目录定位

`src/components/common/` 用于放置页面基础组件，特点是：

1. 高频复用
2. 只承载结构和通用交互
3. 不直接绑定强业务语义
4. 不直接依赖具体模块的数据结构

当前纳入该目录的组件包括：

1. `PageHeader`
2. `PageToolbar`
3. `BaseModal`
4. `FormModal`
5. `ConfirmModal`
6. `LoadingState`
7. `ErrorState`

## 2. PageHeader 使用规范

适合：

1. 列表页
2. 详情页
3. 设置页

支持：

1. `title`
2. `description`
3. `backButton`
4. `actions`
5. `extra`
6. `children`

说明：

1. `actions` 适合放页面级操作按钮
2. `backButton` 适合详情返回或局部返回
3. 不应把复杂业务工具区硬塞进 `PageHeader`

## 3. PageToolbar 使用规范

`PageToolbar` 保持插槽式容器设计，不做复杂配置驱动。

推荐使用：

1. `search`：搜索框
2. `filters`：筛选器、状态筛选、时间筛选
3. `actions`：右侧轻量操作按钮
4. `batchActions`：批量操作容器
5. `children`：特殊布局补充

说明：

1. `PageToolbar` 负责布局，不负责筛选逻辑
2. 查询参数、排序规则和筛选条件仍由页面自己管理

## 4. Modal 职责边界

### BaseModal

职责：

1. 统一遮罩
2. 统一容器
3. 统一标题区、内容区、底部区

不负责：

1. 字段定义
2. 表单校验
3. 危险操作语义

### FormModal

职责：

1. 为新增、编辑、设置类弹窗提供统一外壳
2. 提供默认提交/取消按钮区

不负责：

1. 表单字段结构
2. 表单校验
3. 业务提交流程

### ConfirmModal

职责：

1. 为删除、清空、禁用、重置等操作提供统一确认交互

不负责：

1. 承载复杂业务流程
2. 替代所有确认交互
3. 实现多步骤确认状态机

## 5. LoadingState 与 SkeletonTable 的边界

`LoadingState` 适合：

1. 页面级加载
2. 区块级加载
3. 行内加载

`SkeletonTable` 适合：

1. 复杂表格
2. 需要稳定表头和行骨架的列表页
3. 排序、选择、批量操作较多的表格场景

原则：

1. 不是所有表格都要强制切到 `LoadingState`
2. 复杂表格优先保留 `SkeletonTable`

## 6. EmptyState 使用规范

适合：

1. 页面级空状态
2. 模块级空状态
3. 搜索无结果

建议：

1. 标题简洁明确
2. 说明聚焦下一步
3. 动作按钮只在确实存在下一步动作时出现

## 7. ErrorState 使用规范

适合：

1. 页面级加载失败
2. 无权限
3. 未找到

不适合：

1. 普通按钮点击失败
2. 简单保存失败
3. 细粒度字段校验错误

说明：

1. 普通操作失败可以继续使用 toast
2. `ErrorState` 不替代 `ErrorBoundary`

## 8. StatusBadge 适用范围

适合：

1. 通用状态展示
2. 常量映射明确的状态

不适合：

1. 流程进度条
2. 平台色展示
3. 强业务交互状态
4. 与动作强绑定的特殊标签

原则：

1. 统一优先
2. 但不要为了统一而破坏语义

## 9. BatchActions 定位

`BatchActions` 是业务通用组件，不是页面基础组件。

原因：

1. 它直接绑定“选中项”“批量审批”“批量删除”等业务语义
2. 适合在多个业务列表页复用
3. 不应归入只负责结构的 `common` 层

## 10. 当前阶段不建议继续抽象的内容

当前不建议继续抽象：

1. 流程进度条
2. 批量操作内部逻辑
3. 实时同步高亮
4. 详情页业务区块
5. 表格列定义
6. 状态颜色专用映射
7. 排序逻辑
8. 时间快捷填充逻辑
9. 图表和统计计算逻辑
10. 节点编辑器
11. 编辑器相关区域

核心原则：

页面外层统一优先，业务内核保守处理。

## 11. ContentEditor 统一编辑器入口规范

`ContentEditor` 是 P1-002B 后业务页面推荐使用的统一编辑器入口，位置为 `src/components/ContentEditor.tsx`。

定位：

1. `ContentEditor` 负责收敛业务页面对编辑器的引用入口。
2. `src/components/editor/Editor.tsx` 是当前长期主线底层编辑器，基于 Tiptap。
3. `src/components/RichTextEditor.tsx` 是 legacy 兼容编辑器，仅用于旧页面分阶段迁移。
4. 新业务页面禁止新增直接引用 `RichTextEditor` 或 `components/editor/Editor`。
5. 旧页面迁移必须分阶段进行，不允许一次性替换全部引用。

当前统一接口：

1. `value`：HTML string。
2. `onChange`：输出 HTML string。
3. `readOnly`：只读模式。
4. `placeholder`：占位文案。
5. `onSave`：保存回调，由底层主线编辑器触发时透传当前 HTML string。
6. `mode`：`rich`、`legacy`、`readonly`。
7. `minHeight` 和 `className`：由统一入口外层容器承接，不修改底层编辑器内部结构。

P1-002B 约束：

1. 不引入 JSON 存储变更。
2. 不引入 Markdown 存储变更。
3. 不修改数据库、接口和保存逻辑。
4. 不实现协同编辑。
5. 不删除旧编辑器。
6. 不替换现有业务页面引用。

P1-002C 接入状态：

1. `Topics` 创建选题弹窗已通过 `ContentEditor` 接入，当前使用 `mode="legacy"`。
2. `AddTopic` 大纲编辑区已通过 `ContentEditor` 接入，当前使用 `mode="legacy"`。
3. `src/pages/` 下不应再新增 `RichTextEditor` 或 `RichEditor` 的直接引用。
4. 核心页面直接引用 `components/editor/Editor` 属于阶段性存量，后续进入 P1-002D 时逐页迁移。

## 12. 编辑器退役策略

P1-002E-1 后，legacy 编辑器体系进入冻结与退役倒计时状态。本阶段只做规范收敛，不删除旧组件，不改变运行逻辑。

当前状态：

1. `src/components/RichTextEditor.tsx` 已进入 deprecated 状态。
2. `RichTextEditor` 仅允许作为 `ContentEditor mode="legacy"` 的内部兼容实现。
3. 业务页面禁止直接引用 `RichTextEditor`。
4. 业务页面禁止新增 `RichEditor` 等旧编辑器别名。
5. 新页面和新改造页面必须优先引用 `ContentEditor`。
6. 除已存在的兼容入口外，禁止新增 `mode="legacy"` 使用路径。

统一入口：

1. 编辑器统一入口为 `ContentEditor`。
2. `ContentEditor mode="rich"` 是后续主线模式。
3. `ContentEditor mode="legacy"` 仅作为临时兼容层保留。
4. `components/editor/Editor` 是 Tiptap 主线底层实现，不建议业务页面直接引用。

退役目标：

1. 后续逐步评估 `Topics` / `AddTopic` 等 legacy 使用点是否可切换到 `mode="rich"`。
2. 当业务页面不再使用 `mode="legacy"` 后，再评估删除 `RichTextEditor`。
3. 删除旧编辑器前必须确认历史 HTML、批注、样式和保存链路均可接受。

## 13. 业务组件拆分原则

后续组件治理允许中等规模拆分，但应区分 `common` 基础组件和业务组件，避免把强业务逻辑放入公共组件，也避免为了形式复用而过度抽象。

### common 组件

`common` 组件用于页面基础结构和通用交互，例如：

1. `PageHeader`
2. `PageToolbar`
3. `BaseModal`
4. `FormModal`
5. `ConfirmModal`
6. `LoadingState`
7. `EmptyState`
8. `ErrorState`

要求：

1. 保持轻量。
2. 保持通用。
3. 不放强业务逻辑。
4. 不直接依赖具体业务模块的数据结构。

### 业务组件

业务组件允许按模块拆分，例如：

1. `WorkflowTemplateList`
2. `WorkflowTemplateForm`
3. `WorkflowNodeEditor`
4. `ProductionEditorPanel`
5. `StageContentEditor`
6. `TopicBasicForm`
7. `PublishingScheduleForm`

要求：

1. 放在对应业务模块目录。
2. 不要强行放入 `common`。
3. 不要为了通用而过度抽象。
4. 业务语义优先于形式复用。
5. 拆分必须服务于当前模块的可读性、可维护性或验证边界。

## 14. WorkflowDesigner 规则提示组件规范

当前 WorkflowDesigner 已接入 Workflow Engine v2 strict。节点编辑区域允许展示规则层和决策层提示，但不得把这些提示抽象进 `common` 基础组件。

当前 UI 能力：

1. 节点 risk badge：显示 `low`、`medium`、`high` 与 confidence。
2. warning hint：提示节点需要复核。
3. required fix hint：strict policy 不通过时提示必须修复。
4. suggested transition：可点击并应用到当前节点 `status_to`。
5. hover tooltip：展示 shadow logs、latest reason、heatmap、risk score 和 decision reason。

组件边界：

1. risk badge、suggestion、decision reason 属于 WorkflowDesigner 业务组件范畴。
2. 不应放入 `src/components/common/`。
3. 不应影响通用 `FormModal`、`ConfirmModal`、`PageHeader` 的职责。
4. strict 阻断属于 workflow policy 业务语义，不属于通用表单校验能力。

## 15. WorkflowDesigner Engine UI 规范

P1-003E 后，WorkflowDesigner 已进入 engine UI 阶段，不再是普通配置页面。它同时承载 decision 提示和 enforcement 强约束两层逻辑。

### UI 状态语义

1. 红色节点：`blocked`，表示 strict policy / decision 不通过，保存时必须阻断。
2. 黄色节点：`warning`，表示 non-strict 或中风险提示，可提示但不等同于强制失败。
3. `risk badge`：显示 `low` / `medium` / `high`。
4. `suggestion button`：可点击应用建议流转，但不自动保存。
5. hover 信息：可展示 shadow log count、latest reason、heatmap、risk score、decision reason。

### Decision 与 Enforcement 边界

1. Decision Layer 负责生成风险、原因、置信度和建议流转。
2. Enforcement Layer 负责在 `v2_strict` 下阻断非法配置或非法审批。
3. UI 不应自行发明与 `policy.ts` 不一致的规则。
4. UI 可以展示建议，但不得绕过 strict gate。

### 组件抽象约束

1. WorkflowDesigner 的 risk badge、warning hint、required fix、suggestion button 属于业务组件能力。
2. 不应下沉到 `common` 通用组件。
3. 不应复用为普通表单校验规则。
4. 不应影响 `PageHeader`、`PageToolbar`、`FormModal`、`ConfirmModal` 等基础组件职责。
