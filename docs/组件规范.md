## 1. 组件分层原则

### 已实现

从当前目录结构可总结出初步分层：

1. 布局层组件
- `Layout`
- `Sidebar`
- `ProtectedRoute`

2. 基础反馈组件
- `Skeleton`
- `EmptyState`
- `ErrorBoundary`
- `RealtimeToast`

3. 业务通用组件
- `Pagination`
- `BatchActions`
- `AnnouncementBoard`
- `DataExport`
- `StatCard`

4. 编辑器域组件
- `Editor` 及其工具栏、菜单、目录、扩展

5. 登录域组件
- `EnterpriseLoginLayout`

### 规范建议

1. 基础层
- 不承载具体业务实体

2. 业务层
- 可依赖业务字段，但尽量保持多页面复用

3. 场景层
- 可绑定特定页面或特定模块

4. 编辑器域
- 作为单独能力域维护，不与普通表单组件混写

## 2. 基础组件规范

### 已实现

当前可明确归为基础组件的有：

- `EmptyState`
- `Skeleton` 系列
- `Badge` 中的状态标签类组件
- `ErrorBoundary`
- `Pagination`

### 规范建议

1. 基础组件应尽量通过 props 控制显示，而不是内置过多业务逻辑
2. 基础组件应优先依赖主题样式而不是页面局部硬编码
3. 基础组件要优先保证跨页面复用

## 3. 业务组件规范

### 已实现

当前业务组件主要包括：

- `AnnouncementBoard`
- `BatchActions`
- `DataExport`
- `PomodoroTimer`
- `TopicSpinner`
- `WeeklyReport`
- `CollaborationHeatmap`
- `UpdateNotification`

### 规范建议

1. 业务组件允许包含模块语义
2. 业务组件应避免直接写死页面容器布局
3. 如果某业务组件被多个页面共享，应继续保留组件形态
4. 如果某组件只服务单页且高度耦合，可标注为场景组件

## 4. 编辑器组件规范

### 已实现

当前编辑器组件体系是项目中最完整的一组组件能力之一，真实包含：

- 主编辑器 `Editor`
- 工具栏 `Toolbar`
- 气泡菜单 `BubbleMenu`
- 右键菜单 `ContextMenu`
- 浮动菜单 `FloatingMenu`
- Markdown 预览 `MarkdownPreview`
- 目录面板 `TableOfContents`
- 批注扩展 `CommentExtension`

### 规范建议

1. 编辑器域组件保持单独目录，不混入普通通用组件目录
2. 编辑器域组件之间允许高内聚协作
3. 编辑器状态、批注、版本、保存反馈属于同一能力域，应统一治理
4. 编辑器域不建议拆得过碎，以免破坏当前已经形成的功能闭环

## 5. 表格组件规范

### 已实现

当前项目没有独立的统一表格组件，但已存在真实相关能力：

- `Pagination`
- `BatchActions`
- `SkeletonTable`
- `Badge` 中状态标签
- 列表页内表格结构模式

### 规划中

建议后续抽离但当前尚不存在的统一组件：

- `TableToolbar`
- `TableEmptyState`
- `TableStatusCell`
- `TableActionCell`

### 规范建议

1. 先保留页面内表格结构
2. 对重复出现的表头、操作列、状态列样式逐步提炼
3. 不要一次性强推全局表格重构

## 6. 表单组件规范

### 已实现

当前项目中存在真实表单能力，但未形成完整独立表单组件库。

已确认存在的相关组件：

- `RichTextEditor`
- 编辑器 `Editor`
- 登录页输入结构
- 各页面内输入框、选择器、日期选择等原生控件组合

### 规划中

建议后续统一但当前尚不存在的抽象：

- `FormSection`
- `FormField`
- `FormActions`
- `ConfirmDialog`

### 规范建议

1. 普通字段优先使用统一输入样式
2. 富文本字段继续使用独立编辑器组件
3. 页面中重复出现的表单区块可逐步抽离，而不是先引入复杂表单框架

## 7. 弹窗组件规范

### 已实现

当前项目中存在多处弹窗用法，但尚未看到统一通用弹窗组件。

真实存在的弹窗形态：

- 删除确认弹窗
- 新建选题弹层
- 登录首次改密弹窗
- 编辑器局部浮层和菜单

### 规划中

建议后续统一但当前尚不存在的能力：

- 通用 `Modal`
- `ConfirmModal`
- `FormModal`

### 规范建议

1. 确认类弹窗应统一结构
2. 表单类弹窗应统一头部、内容区、操作区
3. 不同页面不要重复造相似弹窗壳子

## 8. 状态组件规范

### 已实现

当前真实状态组件与模式包括：

- `StatusBadge`
- `RarityBadge`
- `PointBadge`
- `EmptyState`
- `SkeletonLine`
- `SkeletonAvatar`
- `SkeletonCard`
- `SkeletonTable`
- `RealtimeToast`
- `ErrorBoundary`

### 规范建议

1. 状态标签继续保留统一出口
2. 加载态优先使用骨架组件
3. 空态优先使用 `EmptyState`
4. 全局错误保护继续由 `ErrorBoundary` 负责

## 9. 组件命名规范

### 已实现

当前组件命名整体较清晰，主要特征为：

1. PascalCase 文件名
2. 组件名以职责命名
3. 编辑器组件以能力命名
4. 页面级辅助组件没有过度缩写

### 规范建议

1. 通用组件使用功能名
- 如 `Pagination`、`Sidebar`

2. 业务组件使用业务语义名
- 如 `AnnouncementBoard`、`PomodoroTimer`

3. 不要用模糊名字
- 避免 `CommonBox`、`DataThing` 之类命名

4. 页面内部若后续抽离组件，应使用“模块 + 角色”的命名方式

## 10. 组件复用原则

### 规范建议

1. 同一视觉模式出现 3 次以上，优先考虑抽离
2. 状态标签、空态、加载态优先统一
3. 不因一次使用就过早抽象
4. 高耦合编辑器逻辑优先留在编辑器域内部
5. 优先复用现有组件，而不是在页面中复制结构

## 11. 哪些组件建议保留

### 已实现且建议保留

1. `Layout`
2. `Sidebar`
3. `ProtectedRoute`
4. `Pagination`
5. `EmptyState`
6. `Skeleton` 系列
7. `Badge` 系列状态标签
8. `RealtimeToast`
9. `CommandPalette`
10. `KeyboardHelp`
11. `Editor` 及其整套编辑器组件
12. `AnnouncementBoard`
13. `PomodoroTimer`
14. `StatCard`

保留原因：

- 已形成复用价值
- 已具备清晰职责
- 后续仍适合作为设计系统和协作平台的基础资产

## 12. 哪些组件建议后续统一或重构

### 需要统一风格或结构

1. 各页面中的表格工具栏结构
2. 各页面中的筛选与搜索输入区
3. 各页面中的弹窗壳子
4. 页面内自定义按钮样式组合
5. 各类状态标签调用方式

### 需要进一步核查职责边界

1. `RichTextEditor`
- 当前与 `editor/Editor` 并存，需明确一个偏旧、一个偏新，后续应梳理长期定位

2. `ContextMenu.tsx` 与 `editor/ContextMenu.tsx`
- 当前同名但属于不同层级和不同职责，后续需要在文档和命名上进一步区分

3. `WeeklyReport`
- 需核查其是否已成为稳定业务组件，还是阶段性功能组件

4. `CollaborationHeatmap`
- 需核查其当前页面使用情况和长期产品价值

### 规划中

建议未来抽离的统一组件：

- 通用 `Modal`
- 通用 `PageHeader`
- 通用 `FilterBar`
- 通用 `DataTableShell`
- 通用 `FormField`

## 结论

当前组件体系已经有不错的基础，尤其在：

1. 布局组件
2. 状态反馈组件
3. 编辑器组件
4. 部分业务辅助组件

后续不建议大拆大改，建议遵循：

1. 先保留已有高价值组件
2. 再统一高重复页面结构
3. 最后逐步建立更稳定的组件分层
