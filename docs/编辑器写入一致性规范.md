# 编辑器写入一致性规范

## 三源模型

- Yjs = realtime truth：多人协作期间的实时唯一状态，以 runtime Y.Doc 为准。
- Database = persistence truth：`production.content` 与 `shooting.script_content` 只负责持久化落盘。
- Snapshot = recovery truth：snapshot 只用于 runtime 恢复，不直接覆盖数据库内容。

## 单写原则

所有编辑器内容写入必须经过 `writeConsistency`：

- `syncToYjs()`：负责 database 或 snapshot 到 Yjs runtime 的初始化/恢复入口。
- `syncToDatabase()`：负责 Yjs/UI 变化后的 debounce 持久化写回。
- `defineWriteSource()`：负责声明当前文档的写入来源状态。

编辑器页面可以更新本地 UI state，但不得自行维护另一套内容写入定时器。

## 禁止行为

- 禁止页面直接实现 `production.content` / `shooting.script_content` 的 debounce 写库。
- 禁止绕过 `syncToYjs()` 将 database 内容或 snapshot 内容写入 Y.Doc。
- 禁止 snapshot 直接覆盖 database。
- 禁止把 database 当作实时协作源。
- 禁止把 Yjs runtime 当作持久存储。

## 当前路径

初始化：

1. server runtime/snapshot state 同步到 Yjs。
2. 如果 Y.Doc 为空且 database content 存在，通过 `syncToYjs()` 注入一次。
3. Tiptap 绑定 Yjs fragment。

写回：

1. 用户编辑更新 Yjs/UI。
2. 页面接收 HTML 变化。
3. `syncToDatabase()` 统一 debounce 写回对应字段。
