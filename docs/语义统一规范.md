# 语义统一规范

本文档用于冻结 XMT 当前核心语义，避免后续在内容字段、工作流事实来源和编辑器输出格式上继续分叉。

## 内容字段语义

### 唯一正文源

`production.content` 是当前系统唯一可靠的创作正文源。

1. 数据格式为 HTML string。
2. 编辑器写入和页面保存均应围绕该字段理解。
3. `content_markdown` 与 `content_json` 不得反向覆盖 `production.content`。

### Topic 字段

`topics.outline` 是选题大纲 / 剧本大纲字段，可作为 Production 初始内容来源。

`topics.description` 是选题描述字段，用于表达背景、目标、说明和补充信息。

二者均属于选题阶段内容，不是 Production 创建后的实时引用源。

### Shooting 字段

`shooting.script_content` 是拍摄阶段本地剧本内容。

1. 有本字段时优先使用本阶段内容。
2. 字段为空时，页面可按既有逻辑 fallback 到 Production 内容。
3. 保存后只写 Shooting 本阶段，不回写 Production。

### Publishing 字段

`publishing.script_content` 是发布阶段本地发布内容。

1. 有本字段时优先使用发布阶段内容。
2. 字段为空时，页面可按既有逻辑 fallback 到前序阶段内容。
3. 保存后只写 Publishing 本阶段，不回写 Shooting 或 Production。

## 非主数据源字段

以下字段均为非主数据源，禁止作为 canonical：

1. `content_markdown`
2. `content_json`
3. `outline_markdown`
4. `outline_json`

这些字段当前只可视为兼容、过渡或冗余字段。后续如需升级为可靠结构化内容，必须单独设计迁移、回填、校验和回滚策略。

## Workflow 事实类型

当前系统中的 workflow 事实来源分为四类：

1. `workflow_shadow_logs` = 观测事实，用于记录 policy / decision / strict gate 相关运行上下文。
2. `approval_records` = 业务事实，用于表达审批业务结果和历史记录。
3. `decision engine` = 推导事实，用于从观测数据中推导风险、建议和置信度。
4. `policy.ts` = 规则事实，用于表达当前允许的状态转换、审批人规则和配置规则。

## 统一原则

1. 内容字段以 HTML string 为当前主格式。
2. Workflow 观测事实不替代业务事实。
3. 推导事实不替代规则事实。
4. 规则事实不直接改写历史业务事实。
5. 后续治理必须先明确语义来源，再修改实现。

## Collaboration 语义

`collaborationKey` 是编辑器协作文档的运行时唯一标识，不是数据库主键字段，也不新增数据库结构。

当前约定：

1. 创作管理编辑器：`production:{productionId}`，最终保存到 `production.content`。
2. 成片制作编辑器：`shooting:{shootingId}`，最终保存到 `shooting.script_content`。
3. realtime editing is ephemeral state：在线用户、光标、选区、typing、Yjs 增量同步状态只存在于运行时。
4. 数据库仍只保存业务最终内容，不保存协作 presence、awareness 或临时 CRDT room 状态。
5. 冲突处理以 CRDT merge 为主，断线后的最终业务落库遵循现有 API 的 last write wins 行为。
