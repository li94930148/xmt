# 重构规划

## 文档说明

本规划结合当前项目真实结构、现有文档和未来目标整理。

重构不是当前立刻执行的大动作，而是一条长期治理路线。目标不是推翻现有系统，而是在保留已有价值的基础上，把 XMT 从内容生产中台稳步升级为品牌中台，再向行业通用系统扩展。

重构总原则：

1. 先文档化
2. 再局部治理
3. 再模块重构
4. 最后才考虑推翻重做

## 1. 当前架构现状

当前 XMT 是一个前后端同仓的单体全栈项目：

1. 前端：React + TypeScript + Vite
2. 后端：Express + TypeScript
3. 实时通信：Socket.IO
4. 数据库：SQLite + `@libsql/client`

系统已经具备：

1. 内容生产主流程
2. 权限和角色体系
3. 编辑器能力
4. 多个辅助运营模块
5. 工作流与模板雏形

## 2. 当前主要问题

1. 文档、代码、数据库之间需要持续统一口径
2. 前端页面内重复结构较多
3. 弹窗、表格、表单尚未统一标准化
4. 编辑器主路径与历史路径并存
5. 后端仍偏路由驱动
6. 历史结构和兼容结构仍需核查

## 3. 哪些能力应该保留

建议明确保留：

1. 前后端同仓单体结构
2. 选题到分析的主流程模型
3. 权限与角色基础模型
4. 工作流与模板能力
5. 编辑器与创作协作方向
6. 通知与消息联动
7. 当前文档体系本身

## 4. 哪些模块需要重构

1. 列表页结构
2. 详情页结构
3. 页面内联弹窗体系
4. 页面内联表单体系
5. 编辑器路线
6. 后端服务层边界
7. 品牌配置与系统配置能力

## 5. 哪些历史结构需要核查

1. `editing` 表及其业务定位
2. `RichTextEditor` 与 `editor/Editor` 的长期关系
3. 登录页多套布局和多风格实现
4. 历史遗留字段与运行时补字段逻辑
5. 非主路径模块是否仍承担真实业务

## 6. V1 小修小补阶段

### 目标

把当前系统做稳，完成治理基础。

### 重点

1. 文档统一
2. UI 规范统一
3. 组件规范统一
4. 数据库结构说明统一
5. 页面结构标准化

## 7. V1.5 协作增强阶段

### 目标

把系统从“能用的内容中台”升级为“有品牌感、可复制的品牌中台”。

### 重点

1. 品牌配置能力
2. 评论和批注增强
3. 版本和保存反馈增强
4. 工作流与通知联动增强
5. 管理配置能力增强

## 8. V2 协同编辑平台阶段

### 目标

把编辑器和协作能力提升为平台核心。

### 重点

1. 稳定编辑器主路径
2. 强化版本与协作模型
3. 强化实时同步能力
4. 强化内容、流程、评论、消息联动

## 9. V3 传媒公司中台阶段

### 目标

把 XMT 扩展为可服务更完整传媒组织的中台系统。

### 重点

1. 多团队
2. 多业务线
3. 多模板
4. 更完整的运营与数据分析
5. 更明确的治理和审计能力

## 10. 未来如果推翻重做，哪些逻辑应该继承

建议继承：

1. 内容生产主流程认知
2. 权限与角色模型思路
3. 工作流模板思路
4. 编辑器协作方向
5. 消息与流程联动思路
6. 文档治理方法

## 11. 哪些逻辑不建议继承

1. 页面内过重的业务逻辑堆叠
2. 过多硬编码样式
3. 页面级重复弹窗和表格实现
4. 未明确主次关系的历史并存结构
5. 过度依赖本地存储保存系统级配置的方式

## 12. 数据库迁移建议

1. 建立正式迁移机制
2. 区分建表、补字段、数据回填、索引创建职责
3. 核查历史兼容表与主路径表边界
4. 为品牌配置和平台配置预留统一模型

## 13. 前端架构升级建议

1. 从页面导向逐步升级为业务域导向
2. 统一页面模板
3. 抽离高频重复结构
4. 稳定编辑器主路径
5. 为品牌中台配置能力留出明确位置

## 14. 后端架构升级建议

1. 从路由驱动逐步转向服务驱动
2. 路由层专注参数、权限和响应组织
3. 服务层承接流程推进、通知联动和版本保存
4. 为平台化配置预留清晰边界

## 15. 实时协作升级建议

1. 扩大 Socket 事件覆盖范围
2. 区分消息通知事件与协作同步事件
3. 为多人编辑冲突处理预留机制
4. 为评论、批注、审阅联动建立更明确模型

## 16. 云服务器部署升级建议

1. 保留当前轻量单体部署作为低成本版本
2. 逐步补齐反向代理、HTTPS、日志分离、监控告警
3. 标准化备份策略
4. 为后续多实例或更大规模部署预留空间

## 17. 执行顺序建议

1. 文档治理
2. UI 与组件治理
3. 页面结构治理
4. 编辑器路线治理
5. 后端服务层治理
6. 数据库迁移机制升级
7. 协作能力增强
8. 平台化扩展

## 结论

当前 XMT 最适合走的不是“推翻重做”，而是“边治理边沉淀”的路线：

1. 先把现有能力做稳
2. 再把它做成自己的品牌中台
3. 最后把通用能力抽象出来，扩展为行业可用系统
