# 重构规划

## 文档说明

本文档结合当前项目现状、现有文档、真实前后端结构、前端组件与页面组织方式整理。

本文档不代表立即执行重构，而是作为未来治理的路线文件。

重构总原则：

1. 先文档化
2. 再局部治理
3. 再模块重构
4. 最后才考虑推翻重做

## 1. 当前架构现状

### 已实现

当前 XMT 是一个前后端同仓的单体式全栈项目：

- 前端：React + Vite + TypeScript
- 后端：Express + TypeScript
- 实时通信：Socket.IO
- 数据库：SQLite + `@libsql/client`

前端特征：

- 页面数量较多
- 路由覆盖主要业务模块
- 已形成后台主布局和较完整编辑器能力

后端特征：

- 路由按业务模块拆分
- 中间件已有认证、权限、限流
- 业务逻辑仍较多分布在路由与工具函数中

数据库特征：

- 主流程结构清晰
- 迁移方式偏增量追加
- 物理外键较少，逻辑关联较多

## 2. 当前主要问题

### 已实现现状中暴露的问题

1. 前端模块边界不够稳定
- 页面中承载了较多业务逻辑
- 列表页、详情页有大量重复结构

2. 组件体系有基础但未完全系统化
- 已有复用组件
- 但表格、表单、弹窗、页面头部等尚未完全统一

3. 编辑器能力强，但与普通页面体系差异较大
- 长期需要把“创作工具域”单独视为一级能力

4. 后端仍偏路由驱动
- 服务层和领域层边界不够清楚

5. 数据库治理仍偏快速迭代风格
- 多处依赖应用层维护数据一致性
- 存在历史结构或过渡结构待核查

6. 配置中心化程度不高
- 部分系统设置和界面设置仍依赖本地存储

## 3. 哪些能力应该保留

### 已实现且建议保留

1. 前后端同仓基础结构
- 现阶段仍有较高开发效率

2. 主业务流程模型
- `topics`
- `production`
- `shooting`
- `publishing`
- `analytics`

3. 权限模型基础
- `users`
- `roles`
- `permissions`
- `user_roles`
- `role_permissions`

4. 编辑器能力
- Tiptap 编辑器
- 批注扩展
- 工具栏
- Markdown/HTML/JSON 多形态内容处理

5. 实时通信基础
- 用户鉴权连接
- 消息提醒
- 房间机制

6. 后台主布局体系
- 侧边栏
- 顶部栏
- 内容容器

7. 文档体系本身
- 当前已逐步建立的 `docs/` 应继续作为治理入口

## 4. 哪些模块需要重构

### 建议优先重构

1. 前端列表页模式
- 搜索、筛选、表格、分页、批量操作存在较多重复

2. 详情页模式
- 头部信息区、状态区、操作区结构可统一

3. 设置页与弹窗体系
- 当前已有多种实现方式，建议逐步统一

4. 后端主流程服务
- 选题流转
- 创作保存
- 审批推进
- 通知联动

5. 主题与设计变量调用方式
- 减少直接写死的颜色和局部样式

## 5. 哪些历史结构需要核查

### 已实现但需核查

1. `editing` 表
- 实际数据库中存在
- 但当前前台和路由结构中未见稳定对应模块

2. `RichTextEditor` 与 `editor/Editor`
- 当前并存
- 需确认一个是否为旧实现过渡层

3. 登录页多套视觉风格实现
- 当前登录页已存在多种布局风格逻辑
- 需确认长期是否都要保留

4. 页面中的局部硬编码样式
- 需核查哪些是稳定风格，哪些只是阶段性实现

## 6. V1 小修小补阶段

### 目标

在不推翻架构的前提下，先稳定系统。

### 建议动作

1. 完成文档体系
2. 统一页面级 UI 结构
3. 统一状态标签、空态、骨架态
4. 统一表格工具栏与分页模式
5. 梳理旧结构和重复组件
6. 核查数据库中的历史表与过渡表

## 7. V1.5 协作增强阶段

### 目标

从“流程管理”增强到“流程 + 协作”。

### 建议动作

1. 优化编辑器保存、反馈和状态提示
2. 增强评论、批注、审阅链路
3. 扩展实时同步事件
4. 统一协作提示和通知机制
5. 优化工作流模板与审批体验

## 8. V2 协同编辑平台阶段

### 目标

让 XMT 从内容生产中台升级为协同编辑平台。

### 建议动作

1. 把编辑器能力提升为核心平台能力
2. 建立更清晰的文档协作与版本模型
3. 把内容、评论、审批、消息串成统一上下文
4. 建立模块化前端结构
5. 建立更稳定的后端服务层

## 9. V3 传媒公司中台阶段

### 目标

支撑更大组织和更多业务线。

### 建议动作

1. 支撑多团队多项目
2. 统一资源、账号、内容资产管理
3. 强化经营分析和报表体系
4. 逐步向平台化和服务化演进

## 10. 未来如果推翻重做，哪些逻辑应该继承

### 建议继承

1. 主业务流程抽象
- 选题到发布的流程模型

2. 权限模型思路
- 用户、角色、权限、映射关系

3. 编辑器域经验
- 批注
- 版本
- 工具栏能力
- 富文本与 Markdown 双向处理经验

4. 通知与实时联动思路

5. 文档体系和治理方法

## 11. 哪些逻辑不建议继承

### 建议谨慎继承或不继承

1. 页面内过重业务逻辑组织方式
2. 过多局部硬编码样式
3. 非统一的弹窗和表单实现模式
4. 依赖本地存储保存系统级配置的方式
5. 历史过渡结构未核实前的数据库表设计

## 12. 数据库迁移建议

### 规划中

1. 建立正式迁移版本机制
2. 区分建表、加列、回填、索引创建的迁移职责
3. 对核心业务关系逐步补足外键或替代治理机制
4. 对历史表和未明确使用表先核查再决定保留/归档
5. 对统计类能力逐步规划更稳定的数据聚合方案

## 13. 前端架构升级建议

### 规划中

1. 由页面导向逐步升级为模块导向
2. 每个业务域逐步形成：
- `api`
- `components`
- `hooks`
- `types`
- `pages`

3. 统一页面模板
4. 抽离重复表格、表单、弹窗模式
5. 将主题变量和设计规范进一步系统化

## 14. 后端架构升级建议

### 规划中

1. 从路由驱动逐步转向服务驱动
2. 路由层只负责：
- 参数解析
- 权限验证
- 响应格式

3. 服务层负责：
- 流程推进
- 版本保存
- 消息联动
- 审批规则

4. 后续可视情况增加仓储层或领域层

## 15. 实时协作升级建议

### 规划中

1. 扩大 Socket 事件覆盖范围
2. 为协作编辑建立更细粒度事件模型
3. 区分消息提醒事件与数据同步事件
4. 为多人编辑冲突处理预留机制
5. 为协作审阅建立更清晰的事件订阅模式

## 16. 云服务器部署升级建议

### 规划中

1. 继续保留当前单体部署作为低成本版本
2. 生产环境逐步补齐：
- 反向代理
- HTTPS
- 进程守护
- 日志分离
- 监控告警

3. 对数据库备份策略进一步标准化
4. 为后续服务拆分预留网络和配置结构

## 17. 重构执行顺序建议

### 建议顺序

1. 文档治理
2. UI 与组件治理
3. 列表页和详情页结构统一
4. 编辑器域能力巩固
5. 后端服务层下沉
6. 数据库迁移机制升级
7. 实时协作增强
8. 平台化扩展

## 结论

当前 XMT 不适合直接推翻重做，更适合：

1. 先把已有能力文档化
2. 再把高重复结构标准化
3. 再按业务域做局部重构
4. 在协作平台目标下逐步升级

最应该被保护下来的，不是某个页面细节，而是：

1. 内容生产流程认知
2. 权限与治理基础
3. 编辑器与创作协作能力
4. 已形成的真实业务数据模型
