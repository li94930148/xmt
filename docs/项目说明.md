# 项目说明

## 项目简介

XMT 是一个面向内容团队的内容生产中台项目，当前以“选题立项 -> 创作编辑 -> 拍摄排期 -> 发布执行 -> 数据复盘”为主链路，帮助团队把分散的内容工作统一到同一套系统中管理。

项目目前已经具备完整的中后台产品雏形，不再只是单点工具，而是在朝“品牌化内容中台”方向持续演进。

## 项目定位

### 当前定位

当前 XMT 的定位是：

1. 内容生产团队内部使用的工作中台
2. 适合小到中型内容团队统一流程、统一数据、统一协作入口
3. 兼顾流程管理、创作编辑、协作提醒、权限治理和数据复盘

### 中期定位

中期目标是把 XMT 打造成我们自己的品牌中台产品，具备以下特征：

1. 可品牌化配置
2. 可承载多个内容团队或业务线
3. 可复制到不同公司内部场景
4. 具备标准化实施和交付能力

### 长期定位

长期目标是从“自用品牌中台”升级为“可通用于行业的内容协作与生产系统”，逐步向更广义的行业中台演进。

## 用户角色

基于当前代码和权限结构，系统已经覆盖以下真实角色：

1. 管理员 `admin`
   - 系统管理、权限配置、备份、品牌设置、全局治理
2. 编导 `director`
   - 审核、流程推进、拍摄发布协调、数据查看
3. 成员 `member`
   - 选题创建、内容编辑、评论协作、基础业务执行
4. 编辑 `editor`
   - 内容编辑、创作阶段协作、局部流程执行

未来规划中还可进一步扩展为：

1. 品牌管理员
2. 业务线负责人
3. 客户/外部协作方
4. 行业模板管理员

## 长期发展目标

XMT 的长期目标不是只做一个内部管理后台，而是沉淀为一套可产品化、可品牌化、可行业扩展的系统能力。

发展路线分为三层：

1. 第一层：把当前内容生产中台做稳
   - 流程稳定
   - 数据清晰
   - 权限可控
   - 协作顺畅

2. 第二层：做成我们自己的品牌中台
   - 品牌配置
   - 组织与团队扩展
   - 模板化交付
   - 平台化治理

3. 第三层：抽象成行业通用平台
   - 支持更多行业工作流
   - 支持多组织、多品牌、多场景复用
   - 从内容生产中台扩展为通用协作生产系统
## Workflow Engine 当前状态

当前仓库已经具备 Workflow Engine v2 的完整治理链路，真实运行状态为 `v2_strict`，而不是仅停留在提示或观测阶段。

已落地能力包括：

1. Shadow Logging Layer：审批相关运行上下文写入 `workflow_shadow_logs`。
2. Analytics Layer：通过 `/api/workflow/shadow-analytics` 汇总流转频率、节点活跃度、审批人分布和风险评分。
3. Decision Layer：通过 `api/services/workflow_decision_engine.ts` 将观测数据转化为 warning、risk、suggested transition 等决策信息。
4. Partial Control Layer：WorkflowDesigner 展示风险标识、原因提示和建议流转，审批 API 可返回 decision metadata。
5. Strict Control Layer：`WORKFLOW_ENGINE_MODE = 'v2_strict'` 时，审批 API 会在策略不通过时返回阻断结果，WorkflowDesigner 会阻止保存非法节点配置。

需要明确的是，Workflow Engine v2 并未修改工作流数据库主 schema，也未删除 Shadow / Analytics / Decision 能力。当前策略是：在保留可观测和可解释能力的基础上，对明确非法的状态流转和审批执行进行强约束。

相关入口：

1. 规则层：`shared/workflow/policy.ts`
2. 决策层：`api/services/workflow_decision_engine.ts`
3. 审批接入点：`api/routes/workflow-templates.ts`
4. 观测 API：`api/routes/workflow.ts`
5. UI 接入点：`src/pages/WorkflowDesigner.tsx`
